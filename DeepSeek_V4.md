# DeepSeek-V4：迈向高效百万 Token 上下文智能

**DeepSeek-AI**

research@deepseek.com

## 摘要

我们发布了 DeepSeek-V4 系列的预览版本，包括两个强大的混合专家模型（MoE）语言模型——DeepSeek-V4-Pro（1.6T 参数，49B 激活）和 DeepSeek-V4-Flash（284B 参数，13B 激活）——两者均支持一百万 token 的上下文长度。DeepSeek-V4 系列在架构和优化方面引入了若干关键升级：(1) 混合注意力架构，结合压缩稀疏注意力（CSA）和重度压缩注意力（HCA），以提升长上下文效率；(2) 流形约束超连接（mHC），增强传统残差连接；(3) Muon 优化器，实现更快的收敛和更高的训练稳定性。我们在超过 32T 多样化高质量 token 上预训练了两个模型，随后通过全面的后训练流程进一步释放和增强其能力。DeepSeek-V4-Pro-Max 作为 DeepSeek-V4-Pro 的最大推理努力模式，重新定义了开源模型的最先进水平，在核心任务上超越了前代模型。同时，DeepSeek-V4 系列在长上下文场景中具有极高的效率。在一百万 token 上下文设置下，DeepSeek-V4-Pro 仅需 DeepSeek-V3.2 单 token 推理 FLOPs 的 27% 和 KV 缓存的 10%。这使我们能够常规支持一百万 token 上下文，从而让长程任务和进一步的测试时扩展变得更加可行。模型检查点可在 https://huggingface.co/collections/deepseek-ai/deepseek-v4 获取。

## 1 引言

推理模型（DeepSeek-AI, 2025; OpenAI, 2024c）的出现确立了测试时扩展的新范式，为大型语言模型（LLM）带来了显著的性能提升。然而，这种扩展范式从根本上受到标准注意力机制（Vaswani et al., 2017）二次计算复杂度的约束，这为超长上下文和推理过程带来了 prohibitive 瓶颈。同时，长程场景和任务的出现——从复杂的智能体工作流到大规模跨文档分析——也使得高效支持超长上下文成为未来发展的关键。

为了打破超长上下文中的效率壁垒，我们开发了 DeepSeek-V4 系列，包括 DeepSeek-V4-Pro（1.6T 参数，49B 激活）和 DeepSeek-V4-Flash（284B 参数，13B 激活）的预览版本。通过架构创新，DeepSeek-V4 系列在处理超长序列的计算效率上实现了飞跃性提升。这一突破使得高效支持一百万 token 的上下文长度成为可能，为下一代 LLM 开启了百万长度上下文的新时代。我们相信，高效处理超长序列的能力将解锁测试时扩展的下一个前沿，为长程任务的深入研究铺平道路，并为探索在线学习等未来范式奠定必要基础。

与 DeepSeek-V3 架构（DeepSeek-AI, 2024）相比，DeepSeek-V4 系列保留了 DeepSeekMoE 框架（Dai et al., 2024）和多 Token 预测（MTP）策略，同时在架构和优化方面引入了若干关键创新。为提升长上下文效率，我们设计了结合压缩稀疏注意力（CSA）和重度压缩注意力（HCA）的混合注意力机制。CSA 沿序列维度压缩 KV 缓存，然后执行 DeepSeek 稀疏注意力（DSA）（DeepSeek-AI, 2025），而 HCA 对 KV 缓存应用更激进的压缩但保持稠密注意力。为增强建模能力，我们引入了流形约束超连接（mHC）（Xie et al., 2026），升级传统残差连接。此外，我们将 Muon 优化器（Jordan et al., 2024; Liu et al., 2025）引入 DeepSeek-V4 系列的训练，实现更快的收敛和更高的训练稳定性。

为实现 DeepSeek-V4 系列的高效训练和推理以及高效开发，我们引入了若干基础设施优化。首先，我们为 MoE 模块设计并实现了单一融合内核，完全重叠计算、通信和内存访问。其次，我们采用 TileLang（Wang et al., 2026），一种领域特定语言（DSL），以平衡开发生产力和运行时效率。第三，我们提供高效的批次不变且确定性的内核库，确保训练和推理之间的位级可复现性。第四，对于训练框架，我们扩展了自动微分框架，支持张量级检查点以实现细粒度重计算控制；并通过 Muon 优化器的混合 ZeRO 策略、mHC 的成本效益实现（通过重计算和融合内核）以及管理压缩注意力的两阶段上下文并行来提升训练效率。第五，对于推理框架，我们设计了异构 KV 缓存结构，配合磁盘存储策略以实现高效的共享前缀复用。此外，在后训练阶段，我们引入了针对 MoE 专家权重的 FP4 量化感知训练以及索引器 QK 路径，以减少内存和计算。

通过采用混合 CSA 和 HCA，以及对计算和存储的精度优化，与 DeepSeek-V3.2 相比，DeepSeek-V4 系列实现了显著更低的推理 FLOPs 和大幅减小的 KV 缓存大小，尤其在长上下文设置中。图 1 右侧展示了 DeepSeek-V3.2 和 DeepSeek-V4 系列的估计单 token 推理 FLOPs 和累积 KV 缓存大小。在 1M token 上下文场景中，即使激活参数更多的 DeepSeek-V4-Pro，也仅达到 DeepSeek-V3.2 单 token FLOPs（以等效 FP8 FLOPs 衡量）的 27% 和 KV 缓存大小的 10%。此外，激活参数更少的 DeepSeek-V4-Flash 将效率推向更高：在 1M token 上下文设置下，它仅达到 DeepSeek-V3.2 单 token FLOPs 的 10% 和 KV 缓存大小的 7%。此外，对于 DeepSeek-V4 系列，路由专家参数采用 FP4 精度。虽然 FP4 × FP8 操作的峰值 FLOPs 目前在现有硬件上与 FP8 × FP8 相同，但理论上在未来硬件上可以实现 1/3 的效率提升，这将进一步增强 DeepSeek-V4 系列的效率。

在预训练期间，我们分别在 32T token 上训练 DeepSeek-V4-Flash，在 33T token 上训练 DeepSeek-V4-Pro。预训练后，这两个模型可以原生且高效地支持 1M 长度上下文。在我们的内部评估中，DeepSeek-V4-Flash-Base 已经以其更参数高效的设计在大多数基准测试中超越了 DeepSeek-V3.2-Base。DeepSeek-V4-Pro-Base 进一步扩展了这一优势，为 DeepSeek 基础模型树立了新的性能标准，在推理、编程、长上下文和世界知识任务上实现了全面的优越性。

DeepSeek-V4 系列的后训练流程采用两阶段范式：领域特定专家的独立培养，随后通过策略内蒸馏（Gu et al., 2024; Lu and Lab, 2025）进行统一模型整合。最初，对于每个目标领域——如数学、编程、智能体和指令遵循——单独训练一个专家模型。基础模型首先在高质量领域特定数据上接受监督微调（SFT），以建立基础能力。随后，应用组相对策略优化（GRPO）（DeepSeek-AI, 2025）进行强化学习（RL），在针对特定成功标准定制的奖励模型指导下，进一步优化模型的领域对齐行为。这一阶段产生了一组多样化的专业专家，每个都在其 respective 领域表现出色。最后，为整合这些不同的专业能力，通过策略内蒸馏训练单一统一模型，其中统一模型作为学生，学习优化与教师模型的反向 KL 损失。

### 核心评估结果摘要

- **知识**：在广泛世界知识评估中，DeepSeek-V4-Pro-Max（DeepSeek-V4-Pro 的最大推理努力模式）在 SimpleQA（OpenAI, 2024d）和 Chinese-SimpleQA（He et al., 2024）基准测试中显著优于领先的开源模型。在教育知识方面——通过 MMLU-Pro（Wang et al., 2024b）、HLE（Phan et al., 2025）和 GPQA（Rein et al., 2023）评估——DeepSeek-V4-Pro-Max 对其开源同类显示出微弱领先。DeepSeek-V4-Pro-Max 已显著缩小与领先闭源模型 Gemini-3.1-Pro 的差距，尽管在这些基于知识的评估中仍稍逊一筹。

- **推理**：通过扩展推理 token，DeepSeek-V4-Pro-Max 在标准推理基准测试中相对于 GPT-5.2 和 Gemini-3.0-Pro 表现出优越性能。然而，其性能略低于 GPT-5.4 和 Gemini-3.1-Pro，表明其发展轨迹落后最先进水平的前沿模型约 3 到 6 个月。此外，DeepSeek-V4-Flash-Max 达到与 GPT-5.2 和 Gemini-3.0-Pro 相当的性能，确立了自身作为复杂推理任务的高性价比架构。

- **智能体**：在公共基准测试中，DeepSeek-V4-Pro-Max 与领先的开源模型（如 Kimi-K2.6 和 GLM-5.1）持平，但略逊于前沿闭源模型。在我们的内部评估中，DeepSeek-V4-Pro-Max 优于 Claude Sonnet 4.5，并接近 Opus 4.5 的水平。

- **长上下文**：DeepSeek-V4-Pro-Max 在合成和真实用例中，以 1 百万 token 上下文窗口交付了强劲结果，在学术基准测试中甚至超越了 Gemini-3.1-Pro。

- **DeepSeek-V4-Pro vs. DeepSeek-V4-Flash**：DeepSeek-V4-Flash-Max 由于参数规模较小，在知识评估中表现较低。然而，在分配更大思考预算时，它在推理任务上取得了可比较的结果。在智能体评估中，虽然 DeepSeek-V4-Flash-Max 在若干基准测试中与 DeepSeek-V4-Pro-Max 性能匹配，但在更复杂、高难度的任务上仍落后于其更大的 counterpart。

## 2 架构

总体而言，DeepSeek-V4 系列保留了 Transformer（Vaswani et al., 2017）架构和多 Token 预测（MTP）模块（DeepSeek-AI, 2024; Gloeckle et al., 2024），同时在 DeepSeek-V3 基础上引入了若干关键升级：(1) 首先，我们引入流形约束超连接（mHC）（Xie et al., 2026）来增强传统残差连接；(2) 其次，我们设计了混合注意力架构，通过压缩稀疏注意力和重度压缩注意力大幅提升长上下文效率；(3) 第三，我们采用 Muon（Jordan et al., 2024; Liu et al., 2025）作为优化器。对于混合专家（MoE）组件，我们仍采用 DeepSeekMoE（Dai et al., 2024）架构，仅对 DeepSeek-V3 进行了 minor 调整。多 Token 预测（MTP）（DeepSeek-AI, 2024; Gloeckle et al., 2024; Li et al., 2024; Qi et al., 2020）配置与 DeepSeek-V3 相同。所有其他未指定的细节遵循 DeepSeek-V3（DeepSeek-AI, 2024）中建立的设置。图 2 展示了 DeepSeek-V4 的整体架构，详情如下所述。

### 2.1 继承自 DeepSeek-V3 的设计

**混合专家模型（MoE）**。与之前的 DeepSeek 系列模型（DeepSeek-AI, 2024; DeepSeek-AI, 2024）一样，DeepSeek-V4 系列也采用 DeepSeekMoE 范式（Dai et al., 2024）用于前馈网络（FFN），设置了细粒度的路由专家和共享专家。与 DeepSeek-V3 不同，我们将计算亲和度分数的激活函数从 Sigmoid(·) 改为 Sqrt(Softplus(·))。对于负载均衡，我们也采用无辅助损失策略（DeepSeek-AI, 2024; Wang et al., 2024a），辅以轻微的序列级平衡损失，防止单个序列内的极端不平衡。对于 DeepSeek-V4，我们移除了对路由目标节点数量的约束，并仔细重新设计了并行策略以保持训练效率。此外，与 DeepSeek-V3 相比，我们将初始若干 Transformer 块中的稠密 FFN 层替换为采用哈希路由（Roller et al., 2021）的 MoE 层。哈希路由策略根据预定义的哈希函数，针对输入 token ID 确定每个 token 的目标专家。

**多 Token 预测（MTP）**。与 DeepSeek-V3 一样，DeepSeek-V4 系列也设置了 MTP 模块和目标。鉴于 MTP 策略已在 DeepSeek-V3 中得到验证，我们对 DeepSeek-V4 系列采用相同的策略，不做修改。

### 2.2 流形约束超连接（mHC）

如图 2 所示，DeepSeek-V4 系列引入流形约束超连接（mHC）（Xie et al., 2026）来增强相邻 Transformer 块之间的传统残差连接。与朴素超连接（HC）（Zhu et al., 2025）相比，mHC 的核心思想是将残差映射约束到特定流形上，从而在保持模型表达能力的同时增强跨层信号传播的稳定性。本小节简要介绍标准 HC 并描述我们如何设计 mHC 以实现稳定训练。

**标准超连接**。标准 HC 将残差流的宽度扩展 n_hc 倍。具体而言，残差流的形状从 R^d 扩展到 R^(n_hc × d)，其中 d 是实际层输入的隐藏大小。令 X_l = [x_{l,1}; ...; x_{l,n_hc}]^T ∈ R^(n_hc × d) 为第 l 层之前的残差状态。HC 引入三个线性映射：输入映射 A_l ∈ R^(1×n_hc)，残差变换 B_l ∈ R^(n_hc×n_hc)，以及输出映射 C^l ∈ R^(n_hc×1)。残差状态的更新公式为：

X_{l+1} = B_l X_l + C^l F_l(A_l X_l)  
(1)

其中 F_l 表示第 l 层（例如 MoE 层），其输入和输出形状均为 R^d。注意实际层输入 A_l X_l ∈ R^d 也是 d 维的，因此扩展的残差宽度不影响内部层的设计。HC 将残差宽度与实际隐藏大小解耦，提供了一个补充的扩展轴，且计算开销最小，因为 n_hc 通常远小于隐藏大小 d。然而，尽管 HC 在提升模型性能方面显示出潜力，但我们发现当堆叠多层时，训练会频繁出现数值不稳定性，这阻碍了 HC 的扩展。

**流形约束残差映射**。mHC 的核心创新是将残差映射矩阵 B_l 约束到双随机矩阵流形（Birkhoff 多面体）M 上，从而增强跨层信号传播的稳定性：

B_l ∈ M := { M ∈ R^(n×n) | M 1_n = 1_n, 1_n^T M = 1_n^T, M ≥ 0 }  
(2)

这一约束确保映射矩阵的谱范数 ||B_l||_2 被限制在 1 以内，因此残差变换是非扩张的，这增加了前向传播和反向传播期间的数值稳定性。此外，集合 M 在乘法下封闭，这保证了 mHC 深层堆叠场景中的稳定性。另外，输入变换 A_l 和输出变换 C^l 也通过 Sigmoid 函数约束为非负且有界，以避免信号抵消的风险。

**动态参数化**。三个线性映射的参数是动态生成的，分解为动态（输入相关）分量和静态（输入无关）分量。给定输入 X_l ∈ R^(n_hc × d)，首先将其展平并归一化：X̂_l = RMSNorm(vec(X_l)) ∈ R^(1×n_hc d)。然后，我们遵循传统 HC 生成无约束的原始参数 Ã_l ∈ R^(1×n_hc)，B̃_l ∈ R^(n_hc×n_hc)，和 C̃^l ∈ R^(n_hc×1)：

Ã_l = α_l^pre · (X̂_l W_l^pre)  
B̃_l = α_l^res · (X̂_l W_l^res)  
C̃^l = α_l^post · (X̂_l W_l^post)

其中 W_l^pre ∈ R^(n_hc d × n_hc)，W_l^res ∈ R^(n_hc d × n_hc^2)，W_l^post ∈ R^(n_hc d × n_hc) 是可学习参数，α_l^pre、α_l^res、α_l^post 是用于平衡动态和静态分量的可学习标量。静态分量由额外的可学习参数提供：Ã_l^static ∈ R^(1×n_hc)，B̃_l^static ∈ R^(n_hc×n_hc)，C̃^l_static ∈ R^(n_hc×1)。最终参数为：

A_l = Ã_l + Ã_l^static  
B_l = Proj_M(B̃_l + B̃_l^static)  
C^l = C̃^l + C̃^l_static

其中 Proj_M 表示到双随机矩阵流形 M 的投影。输入变换 A_l 和输出变换 C^l 通过 Sigmoid 函数进一步约束：A_l = Sigmoid(A_l)，C^l = Sigmoid(C^l)。

### 2.3 CSA 与 HCA 混合注意力机制

为提升长上下文效率，DeepSeek-V4 系列采用混合注意力架构，结合压缩稀疏注意力（CSA）和重度压缩注意力（HCA）。CSA 和 HCA 都旨在压缩 KV 缓存以减少内存占用和计算量，但采用不同的压缩策略和注意力模式。

#### 2.3.1 压缩稀疏注意力（CSA）

压缩稀疏注意力（CSA）沿序列维度压缩 KV 缓存，然后执行 DeepSeek 稀疏注意力（DSA）。具体而言，CSA 首先将输入序列划分为固定大小的块，然后对每个块内的 KV 进行压缩，保留关键的注意力信息。压缩后的 KV 缓存随后用于稀疏注意力计算，其中每个查询 token 只关注压缩后的块表示和局部窗口内的 token。

CSA 的压缩策略采用轻量级压缩网络，将每个块的 KV 对压缩为固定数量的代表性向量。这种压缩是动态的，根据块内内容自适应调整压缩比例。压缩后的表示保留了原始序列的全局语义信息，同时大幅减少了 KV 缓存的存储需求。

在注意力计算阶段，CSA 采用分层注意力模式：每个查询 token 首先关注压缩后的块表示以获取全局上下文，然后关注局部窗口内的原始 token 以获取细粒度信息。这种分层设计确保了在长序列上的高效注意力计算，同时保持了注意力质量。

#### 2.3.2 重度压缩注意力（HCA）

重度压缩注意力（HCA）对 KV 缓存应用更激进的压缩，但保持稠密注意力模式。与 CSA 的块级压缩不同，HCA 采用全局压缩策略，将整个序列的 KV 信息压缩为固定数量的锚点向量。

HCA 的压缩网络比 CSA 更深，使用多层感知器（MLP）将序列信息压缩为紧凑的表示。这些锚点向量捕获了序列的全局语义结构，作为注意力计算的"记忆摘要"。

在注意力计算阶段，所有查询 token 都关注这些锚点向量，形成全局稠密注意力。由于锚点数量固定（通常远小于序列长度），注意力计算的复杂度从 O(n²) 降低到 O(n × m)，其中 m 是锚点数量。这种设计特别适合需要全局上下文理解的任务。

#### 2.3.3 其他细节

**位置编码**。DeepSeek-V4 系列采用旋转位置编码（RoPE）（Su et al., 2024），并针对超长上下文进行了优化。我们将基础频率从 DeepSeek-V3 的 10,000 调整为更大的值，以支持长达 1M token 的上下文。具体而言，我们使用 NTK-aware 缩放（bloc97, 2023）和 YaRN（Peng et al., 2023）技术来扩展位置编码的外推能力。

**注意力掩码**。对于 CSA，我们使用因果掩码结合块级稀疏模式。对于 HCA，我们使用全可见掩码，因为锚点向量已经编码了全局信息。在混合架构中，我们根据层深度动态调整 CSA 和 HCA 的比例：浅层更多使用 CSA 以保留局部细节，深层更多使用 HCA 以捕获全局语义。

**KV 缓存管理**。我们实现了分层的 KV 缓存管理系统。对于 CSA，压缩后的块表示存储在 GPU 显存中，原始块可根据需要动态加载。对于 HCA，锚点向量常驻显存，原始 KV 可卸载到主机内存或磁盘。这种分层管理使得 1M token 上下文的 KV 缓存总占用控制在合理范围内。

#### 2.3.4 效率讨论

混合 CSA 和 HCA 的设计在效率和效果之间取得了平衡。表 1 比较了不同注意力机制在 1M token 上下文下的效率指标。

| 注意力机制 | 单 Token FLOPs | KV 缓存大小 | 全局上下文 |
|-----------|---------------|------------|-----------|
| 标准稠密注意力 | 100% | 100% | ✓ |
| CSA | 35% | 15% | △ |
| HCA | 20% | 8% | ✓ |
| CSA + HCA 混合 | 27% | 10% | ✓ |

*表 1：不同注意力机制在 1M token 上下文下的效率比较（相对于标准稠密注意力的百分比）。*

如表所示，CSA + HCA 混合架构在保持全局上下文能力的同时，将单 token FLOPs 降低到 27%，KV 缓存大小降低到 10%。这种效率提升使得在消费级硬件上部署 1M token 上下文模型成为可能。

### 2.4 Muon 优化器

Muon（Jordan et al., 2024; Liu et al., 2025）是一种新兴的优化器，通过正交梯度下降来加速收敛。与 AdamW 等传统优化器不同，Muon 对梯度进行正交化处理，使得参数更新方向更加稳定。

在 DeepSeek-V4 系列的训练中，我们将 Muon 应用于所有线性层的权重更新。具体实现中，我们对 Muon 进行了以下适配：

1. **混合策略**：对于嵌入层和归一化层，我们仍使用 AdamW；对于注意力投影和 FFN 权重，使用 Muon。

2. **学习率调度**：Muon 的学习率采用与 AdamW 不同的调度策略。我们使用预热阶段后接余弦退火，但 Muon 的基础学习率设置为 AdamW 的 2 倍。

3. **正交化频率**：为减少计算开销，我们每 4 步执行一次完整的正交化，中间步骤使用近似的正交梯度。

实验表明，Muon 将 DeepSeek-V4-Pro 的预训练收敛速度提升了约 1.3 倍，同时保持了训练稳定性。

## 3 通用基础设施

### 3.1 专家并行中的细粒度通信-计算重叠

DeepSeek-V4 系列的 MoE 层采用专家并行（EP）来分布路由专家。为隐藏通信延迟，我们设计了细粒度的通信-计算重叠机制。

**重叠策略**。我们将每个专家的前向计算分解为多个微操作：输入投影、专家计算、输出投影。在专家并行设置中，token 路由（all-to-all 通信）与当前层的非专家计算（如 RMSNorm、残差连接）重叠。具体而言，我们使用 CUDA 多流机制：一个流处理通信，另一个流处理计算。

**TileLang 融合内核**。我们使用 TileLang 编写了融合内核，将 all-to-all 通信与专家计算融合为单一内核。这种融合消除了内核启动开销，并允许编译器进行全局优化。融合内核的实现细节如下：

```python
# TileLang 伪代码示例
def fused_moe_kernel(input, expert_weights, routing_indices):
    # 阶段 1：异步 all-to-all 发送
    for i in range(num_local_experts):
        async_send(input[routing_indices == i])
    
    # 阶段 2：本地专家计算（与通信重叠）
    for i in range(num_local_experts):
        local_input = async_receive()
        output = expert_forward(local_input, expert_weights[i])
        async_send_back(output)
    
    # 阶段 3：聚合输出
    final_output = async_receive_all()
    return final_output
```

**性能提升**。通过细粒度重叠，我们将 MoE 层的通信开销从占总时间的 35% 降低到 8%，整体训练吞吐量提升约 1.4 倍。

### 3.2 基于 TileLang 的灵活高效内核开发

TileLang（Wang et al., 2026）是一种用于编写高性能 GPU 内核的领域特定语言（DSL）。我们为 DeepSeek-V4 系列使用 TileLang 开发了所有关键内核。

**内核列表**。以下是我们使用 TileLang 实现的主要内核：

| 内核名称 | 描述 | 性能提升 |
|---------|------|---------|
| fused_shared_ffn | 共享专家 FFN 融合内核 | 1.8× |
| hybrid_attention | CSA/HCA 混合注意力 | 2.3× |
| moe_routing | MoE 路由 + all-to-all | 1.5× |
| rms_norm | RMSNorm 融合 | 1.2× |
| rotary_embedding | 旋转位置编码 | 1.3× |
| kv_cache_compress | KV 缓存压缩 | 3.0× |

*表 2：TileLang 内核及其性能提升（相对于 PyTorch 原生实现）。*

**开发流程**。我们的 TileLang 内核开发遵循以下流程：

1. **原型设计**：在 Python 中使用 TileLang DSL 编写内核逻辑。
2. **编译优化**：TileLang 编译器将 DSL 代码编译为优化的 CUDA 内核。
3. **性能调优**：使用 TileLang 的分析工具识别瓶颈并调整 tile 大小、内存布局等。
4. **集成测试**：将编译后的 .so 文件集成到 Rust 推理引擎中。

**Mega-kernel**。对于关键路径，我们使用 TileLang 的 Mega-kernel 能力将多个操作融合为持久化内核。例如，我们将 RMSNorm → QKV 投影 → 混合注意力 → 输出投影 → 残差连接融合为单一 Mega-kernel，消除了中间结果的内存往返。

### 3.3 高性能批次不变且确定性的内核库

为确保训练和推理之间的位级可复现性，我们开发了批次不变且确定性的内核库。

**批次不变性**。我们的内核在不同批次大小下产生数值相同的结果。这通过以下技术实现：

1. **归一化统计的累积方式**：使用 Welford 在线算法计算均值和方差，避免批次大小相关的数值差异。
2. **求和顺序**：使用 Kahan 求和或成对求和，减少浮点舍入误差。

**确定性**。通过以下措施确保确定性：

1. **原子操作替代**：在 reduce 操作中使用确定性的归约树，而非原子操作。
2. **随机数种子管理**：所有随机操作（如 dropout）使用固定的种子序列。
3. **线程调度固定**：CUDA 线程块的调度顺序固定化。

### 3.4 训练框架

#### 3.4.1 Muon 的高效实现

Muon 优化器的核心操作是对梯度矩阵进行 SVD 或正交化。我们实现了以下优化：

1. **增量正交化**：利用相邻步骤梯度之间的相关性，使用增量 SVD 更新而非完整 SVD。
2. **混合精度正交化**：SVD 在 FP32 中执行，但输入梯度使用 FP16/BF16，减少内存带宽。
3. **异步正交化**：正交化计算与参数更新重叠，使用双缓冲机制。

#### 3.4.2 mHC 的成本效益与内存高效实现

mHC 的动态参数生成引入了额外的计算和内存开销。我们通过以下方式优化：

1. **重计算策略**：mHC 的输入映射参数在反向传播时重计算，而非存储。
2. **融合内核**：将 mHC 的动态参数生成与层计算融合为单一内核。
3. **低秩近似**：对于深层网络，使用低秩近似压缩 mHC 的参数矩阵。

#### 3.4.3 长上下文注意力的上下文并行

对于 1M token 上下文，我们采用两阶段上下文并行：

1. **阶段 1 - 序列并行**：将序列划分为多个片段，分布在不同 GPU 上。每个 GPU 计算本地片段的注意力。
2. **阶段 2 - 全局聚合**：使用 ring attention 机制聚合跨 GPU 的注意力结果。

对于 CSA，压缩后的块表示作为全局共享状态，减少跨 GPU 通信。对于 HCA，锚点向量在所有 GPU 间复制，确保全局一致性。

#### 3.4.4 用于灵活激活检查点的扩展自动微分

我们扩展了 PyTorch 的自动微分框架，支持张量级检查点：

1. **细粒度控制**：可以选择性地检查点任意张量，而非整个层。
2. **内存池管理**：检查点的张量存储在预分配的内存池中，减少分配开销。
3. **选择性重计算**：根据内存压力和计算成本，动态决定是否重计算。

### 3.5 推理框架

#### 3.5.1 KV 缓存结构与管理

我们设计了异构 KV 缓存结构：

1. **GPU 显存层**：存储当前批次和近期批次的 KV 缓存。
2. **主机内存层**：使用 CUDA 固定内存存储历史 KV 缓存。
3. **磁盘层**：使用内存映射文件持久化长期 KV 缓存。

**缓存替换策略**：

- **LFU（最不经常使用）**：用于 GPU 显存层的热点专家缓存。
- **SLRU（分段最近最少使用）**：用于主机内存层的路由专家缓存。
- **异步预取**：根据层间专家激活模式，预测并预取下一层可能需要的专家。

#### 3.5.2 磁盘 KV 缓存存储

为支持热启动和跨会话共享，我们实现了磁盘 KV 缓存存储：

1. **序列化格式**：使用自定义二进制格式存储 KV 缓存，支持快速加载。
2. **压缩**：对磁盘 KV 缓存使用量化压缩（FP8 → INT8），减少存储空间。
3. **索引结构**：使用 B+ 树索引 KV 缓存块，支持快速查找。
4. **共享前缀复用**：对于具有共享前缀的多个请求，只存储一份前缀 KV 缓存。

3. **数值溢出**：FP8 训练中的数值溢出。
   - 解决：动态缩放因子 + 损失缩放自动调整。

2. **编程专家**：
   - SFT 数据：竞赛级编程题解（约 1.5M 样本）
   - RL 奖励：代码通过测试用例 + 执行效率
   - GRPO 训练步数：约 400 步

3. **智能体专家**：
   - SFT 数据：多轮工具使用轨迹（约 1M 样本）
   - RL 奖励：任务完成度 + 工具使用效率
   - GRPO 训练步数：约 300 步

4. **指令遵循专家**：
   - SFT 数据：多样化指令-响应对（约 3M 样本）
   - RL 奖励：响应有用性 + 安全性
   - GRPO 训练步数：约 200 步

#### 5.1.2 策略内蒸馏

在专家训练完成后，我们通过策略内蒸馏（OPD）将各专家的能力整合到统一模型中：

1. **教师模型集合**：使用训练好的各领域专家作为教师。
2. **蒸馏目标**：统一模型学习优化反向 KL 散度：
   
   L_distill = E_{x~D}[ Σ_y π_teacher(y|x) log(π_teacher(y|x) / π_student(y|x)) ]

3. **课程学习**：从简单样本开始，逐步增加难度。
4. **蒸馏步数**：约 1000 步，学习率 1e-5。

### 5.2 后训练基础设施

#### 5.2.1 FP4 量化感知训练

为支持 FP4 推理，我们在后训练阶段引入量化感知训练（QAT）：

1. **量化模拟**：在前向传播中模拟 FP4 量化效果：
   - 权重：FP4 e2m1fn（1 符号位 + 2 指数位 + 1 尾数位）
   - 缩放因子：FP8 e8m0fnu，按行、每 32 列一组

2. **直通估计器（STE）**：反向传播时，量化操作的梯度使用 STE 近似。

3. **微调策略**：先 FP8 微调 500 步，再 FP4 QAT 微调 200 步。

#### 5.2.2 全词汇表 OPD 的高效教师调度

在策略内蒸馏中，我们需要在多个教师模型间调度：

1. **动态调度**：根据当前批次样本的领域特征，动态选择相关教师。
2. **教师缓存**：缓存教师模型的输出，避免重复计算。
3. **异步生成**：教师模型的生成与学生模型的训练异步进行。

#### 5.2.3 可抢占且容错的 Rollout 服务

RL 训练需要大量的 rollout（模型生成）：

1. **可抢占设计**：rollout 任务可被更高优先级的训练任务抢占。
2. **容错机制**：失败的 rollout 自动重试，不影响训练流程。
3. **批量生成**：使用 vLLM 进行高效批量生成，吞吐量提升 5 倍。

#### 5.2.4 百万 Token 上下文的强化学习框架扩展

为支持 1M token 上下文的 RL 训练：

1. **分块 Rollout**：将长上下文分为多个块，逐块生成和评估。
2. **稀疏奖励**：仅在关键位置计算奖励，减少计算量。
3. **内存优化**：使用梯度检查点和激活重计算，支持更大批次。

#### 5.2.5 智能体 AI 的沙箱基础设施

为安全地训练智能体能力：

1. **隔离环境**：每个 rollout 在独立的 Docker 容器中执行。
2. **资源限制**：CPU、内存、网络访问的严格限制。
3. **审计日志**：所有操作记录审计日志，便于分析。
4. **工具白名单**：只允许使用预定义的安全工具。

## 6 结论、局限性与未来方向

### 结论

DeepSeek-V4 系列通过架构创新和基础设施优化，实现了超长上下文的高效处理。主要贡献包括：

1. **混合注意力架构**：CSA 和 HCA 的结合在保持模型能力的同时，将 1M token 上下文的推理成本降低 73%。
2. **流形约束超连接**：mHC 增强了深层网络的训练稳定性，支持更大规模的模型。
3. **Muon 优化器**：加速收敛，提升训练效率。
4. **三级缓存系统**：GPU 显存 + 主机内存 + SSD 的层次化缓存，支持路由专家的流式加载。
5. **高效基础设施**：TileLang 内核、细粒度通信-计算重叠、异构 KV 缓存管理。

### 局限性

1. **长上下文质量**：虽然 1M token 上下文在技术上可行，但在极端长上下文中的信息检索精度仍有提升空间。
2. **多模态**：当前版本仅支持文本，未集成图像、音频等多模态能力。
3. **实时交互**：在极低延迟要求的实时交互场景中，性能仍有优化空间。
4. **硬件依赖**：FP4 推理的优势需要未来硬件支持才能完全发挥。

### 未来方向

1. **更长上下文**：探索 10M+ token 上下文的可行性。
2. **多模态扩展**：集成视觉、音频能力，构建真正的多模态模型。
3. **边缘部署**：优化模型以支持边缘设备部署。
4. **持续学习**：实现在线学习能力，使模型能够从交互中持续改进。
5. **高效微调**：开发更高效的微调方法，降低领域适配成本。
