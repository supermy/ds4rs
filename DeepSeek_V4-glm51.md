## 摘要

我们发布 DeepSeek-V4 系列的预览版本，包括两个强大的混合专家（MoE）语言模型——拥有 1.6T 参数（49B 激活）的 DeepSeek-V4-Pro 和拥有 284B 参数（13B 激活）的 DeepSeek-V4-Flash——两者均支持一百万 token 的上下文长度。DeepSeek-V4 系列在架构和优化方面引入了多项关键升级：（1）混合注意力架构，结合压缩稀疏注意力（CSA）和重度压缩注意力（HCA）以提升长上下文效率；（2）流形约束超连接（mHC），增强传统残差连接；（3）Muon 优化器，实现更快的收敛速度和更高的训练稳定性。我们在超过 32T 多样化高质量 token 上对两个模型进行预训练，随后通过全面的后训练流程解锁并进一步增强其能力。DeepSeek-V4-ProMax 作为 DeepSeek-V4-Pro 的最大推理努力模式，重新定义了开源模型的最先进水平，在核心任务上超越了其前代模型。同时，DeepSeek-V4 系列在长上下文场景中具有极高的效率。在一百万 token 上下文设置下，DeepSeek-V4-Pro 仅需 DeepSeek-V3.2 单 token 推理 FLOPs 的 27% 和 KV cache 的 10%。这使我们能够常规性地支持一百万 token 上下文，从而使长期任务和进一步的测试时扩展更加可行。模型检查点可在 https://huggingface.co/collections/deepseek-ai/deepseek-v4 获取。

## 1 引言

推理模型（DeepSeek-AI, 2025; OpenAI, 2024c）的出现确立了测试时扩展的新范式，为大型语言模型（LLM）带来了显著的性能提升。然而，这一扩展范式从根本上受限于标准注意力机制（Vaswani et al., 2017）的二次计算复杂度，这为超长上下文和推理过程造成了难以逾越的瓶颈。与此同时，长期场景和任务的出现——从复杂的智能体工作流到大规模跨文档分析——也使得对超长上下文的高效支持成为未来发展的关键需求。尽管近期的开源努力（Bai et al., 2025a; DeepSeek-AI, 2024; MiniMax, 2025; Qwen, 2025）推进了通用能力，但在处理超长序列方面这一核心架构低效问题仍然是关键障碍，限制了测试时扩展的进一步收益，也阻碍了对长期场景和任务的进一步探索。

为了突破超长上下文中的效率壁垒，我们开发了 DeepSeek-V4 系列，包括拥有 1.6T 参数（49B 激活）的 DeepSeek-V4-Pro 和拥有 284B 参数（13B 激活）的 DeepSeek-V4-Flash 的预览版本。通过架构创新，DeepSeek-V4 系列在处理超长序列的计算效率方面实现了巨大飞跃。这一突破使得高效支持一百万 token 上下文长度成为可能，开启了下一代 LLM 百万级上下文的新纪元。我们相信，高效处理超长序列的能力将解锁测试时扩展的下一个前沿，为长期任务的深入研究铺平道路，并为探索在线学习等未来范式奠定必要基础。

与 DeepSeek-V3 架构（DeepSeek-AI, 2024）相比，DeepSeek-V4 系列保留了 DeepSeekMoE 框架（Dai et al., 2024）和多 Token 预测（MTP）策略，同时在架构和优化方面引入了多项关键创新。为提升长上下文效率，我们设计了一种结合压缩稀疏注意力（CSA）和重度压缩注意力（HCA）的混合注意力机制。CSA 沿序列维度压缩 KV cache，然后执行 DeepSeek 稀疏注意力（DSA）（DeepSeek-AI, 2025），而 HCA 对 KV cache 施加更激进的压缩但保持稠密注意力。为增强建模能力，我们引入了流形约束超连接（mHC）（Xie et al., 2026），升级了传统残差连接。此外，我们将 Muon（Jordan et al., 2024; Liu et al., 2025）优化器引入 DeepSeek-V4 系列的训练中，实现了更快的收敛速度和更高的训练稳定性。

为了实现 DeepSeek-V4 系列的高效训练和推理以及高效的开发流程，我们引入了多项基础设施优化。首先，我们为 MoE 模块设计并实现了单一融合内核，完全重叠计算、通信和内存访问。其次，我们采用 TileLang（Wang et al., 2026），一种领域特定语言（DSL），以平衡开发效率和运行时效率。第三，我们提供了高效的不变批量且确定性的内核库，以确保训练和推理之间的逐位可复现性。第四，在训练框架方面，我们通过张量级检查点扩展了自动微分框架，实现细粒度的重计算控制；并通过面向 Muon 优化器的混合 ZeRO 策略、通过重计算和融合内核实现的高性价比 mHC 实现，以及管理压缩注意力的两阶段上下文并行策略来提升训练效率。第五，在推理框架方面，我们设计了具有磁盘存储策略的异构 KV cache 结构，以实现高效的共享前缀复用。此外，在后训练阶段，我们对 MoE 专家权重和索引器 QK 路径引入了 FP4 量化感知训练，以减少内存和计算开销。

通过采用混合 CSA 和 HCA，以及计算和存储方面的精度优化，DeepSeek-V4 系列相比 DeepSeek-V3.2 实现了显著更低的推理 FLOPs 和大幅缩减的 KV cache 大小，尤其在长上下文设置中表现突出。图 1 右侧展示了 DeepSeek-V3.2 和 DeepSeek-V4 系列的估计单 token 推理 FLOPs 和累积 KV cache 大小。在 1M token 上下文场景中，即使是拥有更多激活参数的 DeepSeek-V4-Pro，其单 token FLOPs（以等效 FP8 FLOPs 计）也仅为 DeepSeek-V3.2 的 27%，KV cache 大小仅为 10%。此外，激活参数更少的 DeepSeek-V4-Flash 将效率推向更高水平：在 1M token 上下文设置下，其单 token FLOPs 仅为 DeepSeek-V3.2 的 10%，KV cache 大小仅为 7%。另外，对于 DeepSeek-V4 系列，路由专家参数采用 FP4 精度。虽然 FP4 × FP8 运算的峰值 FLOPs 在现有硬件上目前与 FP8 × FP8 相同，但在未来硬件上理论上可以实现 1/3 的效率提升，这将进一步增强 DeepSeek-V4 系列的效率。

在预训练阶段，我们分别在 32T token 上训练 DeepSeek-V4-Flash，在 33T token 上训练 DeepSeek-V4-Pro。预训练完成后，这两个模型能够原生且高效地支持 1M 长度上下文。在我们的内部评估中，DeepSeek-V4-Flash-Base 凭借更高效的参数设计，已在大多数基准测试上超越了 DeepSeek-V3.2-Base。DeepSeek-V4-Pro-Base 进一步扩大了这一优势，在 DeepSeek 基础模型中树立了新的性能标杆，在推理、编程、长上下文和世界知识任务上实现了全面领先。

DeepSeek-V4 系列的后训练流程采用两阶段范式：首先独立培养领域特定专家，然后通过策略蒸馏（on-policy distillation）（Gu et al., 2024; Lu and Lab, 2025）进行统一模型整合。最初，针对每个目标领域——如数学、编程、智能体和指令遵循——独立训练一个单独的专家模型。基础模型首先在高质量、领域特定的数据上进行监督微调（SFT），以建立基础能力。随后，使用组相对策略优化（GRPO）（DeepSeek-AI, 2025）应用强化学习（RL），根据针对特定成功标准定制的奖励模型进一步优化模型的领域对齐行为。这一阶段产生了一组多样化的专业专家，每个专家在各自领域表现出色。最后，为了整合这些不同的专业能力，通过策略蒸馏训练一个统一的模型，其中统一模型作为学生，学习优化与教师模型的逆 KL 散度损失。

### 核心评估结果总结

• 知识：在广泛的世界知识评估中，DeepSeek-V4-Pro-Max（DeepSeek-V4-Pro 的最大推理努力模式）在 SimpleQA（OpenAI, 2024d）和 Chinese-SimpleQA（He et al., 2024）基准测试上显著优于领先的开源模型。在教育知识方面——通过 MMLU-Pro（Wang et al., 2024b）、HLE（Phan et al., 2025）和 GPQA（Rein et al., 2023）评估——DeepSeek-V4-Pro-Max 相较于其开源对照模型略有领先。尽管在这些知识评估中仍然落后，但 DeepSeek-V4-Pro-Max 已显著缩小了与领先闭源模型 Gemini-3.1-Pro 的差距。

• 推理：通过扩展推理 token，DeepSeek-V4-Pro-Max 在标准推理基准测试上表现出优于 GPT-5.2 和 Gemini-3.0-Pro 的性能。然而，其性能略逊于 GPT-5.4 和 Gemini-3.1-Pro，表明其发展轨迹落后最先进前沿模型约 3 至 6 个月。此外，DeepSeek-V4-Flash-Max 达到了与 GPT-5.2 和 Gemini-3.0-Pro 相当的性能，确立了其作为复杂推理任务高性价比架构的地位。

• 智能体：在公开基准测试上，DeepSeek-V4-Pro-Max 与领先的开源模型（如 Kimi-K2.6 和 GLM-5.1）持平，但略逊于前沿闭源模型。在我们的内部评估中，DeepSeek-V4-Pro-Max 超越了 Claude Sonnet 4.5，接近 Opus 4.5 的水平。

• 长上下文：DeepSeek-V4-Pro-Max 在具有一百万 token 上下文窗口的合成和真实用例上取得了强劲的结果，在学术基准测试上甚至超越了 Gemini-3.1-Pro。

• DeepSeek-V4-Pro 与 DeepSeek-V4-Flash 对比：DeepSeek-V4-Flash-Max 由于参数规模较小，在知识评估中表现较低。然而，当分配更大的思考预算时，它在推理任务上取得了相当的结果。在智能体评估中，虽然 DeepSeek-V4-Flash-Max 在多个基准测试上与 DeepSeek-V4-Pro-Max 表现持平，但在更复杂、高难度的任务上仍落后于其更大的对照模型。

## 2 架构

总体而言，DeepSeek-V4 系列保留了 Transformer (Vaswani et al., 2017) 架构和多 Token 预测（MTP）模块 (DeepSeek-AI, 2024; Gloeckle et al., 2024)，同时在 DeepSeek-V3 基础上引入了若干关键升级：（1）首先，我们引入了流形约束超连接（mHC）(Xie et al., 2026) 以增强传统残差连接；（2）其次，我们设计了一种混合注意力架构，通过压缩稀疏注意力（CSA）和重度压缩注意力（HCA）大幅提升了长上下文效率；（3）第三，我们采用 Muon (Jordan et al., 2024; Liu et al., 2025) 作为优化器。对于混合专家（MoE）组件，我们仍然采用 DeepSeekMoE (Dai et al., 2024) 架构，仅相对于 DeepSeek-V3 做了少量调整。多 Token 预测（MTP）(DeepSeek-AI, 2024; Gloeckle et al., 2024; Li et al., 2024; Qi et al., 2020) 的配置与 DeepSeek-V3 完全相同。所有其他未指定的细节均遵循 DeepSeek-V3 (DeepSeek-AI, 2024) 中建立的设置。图 2 展示了 DeepSeek-V4 的整体架构，具体细节如下所述。

### 2.1 继承自 DeepSeek-V3 的设计

混合专家。与之前的 DeepSeek 系列模型 (DeepSeek-AI, 2024; DeepSeek-AI, 2024) 一样，DeepSeek-V4 系列也在前馈网络（FFN）中采用 DeepSeekMoE 范式 (Dai et al., 2024)，设置了细粒度的路由专家和共享专家。与 DeepSeek-V3 不同的是，我们将计算亲和度分数的激活函数从 Sigmoid(·) 改为 Sqrt(Softplus(·))。在负载均衡方面，我们同样采用了无辅助损失策略 (DeepSeek-AI, 2024; Wang et al., 2024a)，并辅以轻微的序列级均衡损失，以防止单个序列内出现极端不平衡。对于 DeepSeek-V4，我们移除了路由目标节点数量的约束，并精心重新设计了并行策略以维持训练效率。此外，与 DeepSeek-V3 相比，我们将初始若干 Transformer 块中的稠密 FFN 层替换为采用哈希路由 (Roller et al., 2021) 的 MoE 层。哈希路由策略根据预定义的哈希函数，以输入 token ID 为参数确定每个 token 的目标专家。

多 Token 预测。与 DeepSeek-V3 一样，DeepSeek-V4 系列也设置了 MTP 模块和目标。鉴于 MTP 策略已在 DeepSeek-V3 中得到验证，我们对 DeepSeek-V4 系列采用相同的策略，不做修改。

### 2.2 流形约束超连接

如图 2 所示，DeepSeek-V4 系列引入了流形约束超连接（mHC）(Xie et al., 2026)，以增强相邻 Transformer 块之间的传统残差连接。与朴素超连接（HC）(Zhu et al., 2025) 相比，mHC 的核心思想是将残差映射约束到特定流形上，从而在保持模型表达力的同时增强跨层信号传播的稳定性。本小节简要介绍标准 HC，并描述我们如何设计 mHC 以实现稳定训练。

标准超连接。标准 HC 将残差流的宽度扩展 n_hc 倍。具体而言，残差流的形状从 R^d 扩展为 R^(n_hc × d)，其中 d 是实际层输入的隐藏维度。设 X_l = [x_{l,1}; ...; x_{l,n_hc}]^T ∈ R^(n_hc × d) 为第 l 层之前的残差状态。HC 引入三个线性映射：输入映射 A_l ∈ R^(1 × n_hc)、残差变换 B_l ∈ R^(n_hc × n_hc) 和输出映射 C_l ∈ R^(n_hc × 1)。残差状态的更新公式为：

X_{l+1} = B_l X_l + C_l F_l(A_l X_l),   (1)

其中 F_l 表示第 l 层（例如 MoE 层），其输入和输出形状均为 R^d。注意，实际层输入 A_l X_l ∈ R^d 也是 d 维的，因此扩展的残差宽度不影响内部层的设计。HC 将残差宽度与实际隐藏维度解耦，提供了一个互补的缩放轴，且计算开销极小，因为 n_hc 通常远小于隐藏维度 d。然而，尽管 HC 在提升模型性能方面展现了潜力，我们发现当堆叠多层时，训练会频繁出现数值不稳定，这阻碍了 HC 的扩展。

流形约束残差映射。mHC 的核心创新是将残差映射矩阵 B_l 约束到双随机矩阵流形（Birkhoff 多面体）M 上，从而增强跨层信号传播的稳定性：

B_l ∈ M ≔ { M ∈ R^(n × n) | M 1_n = 1_n, 1_n^T M = 1_n^T, M ≥ 0}.   (2)

此约束确保映射矩阵的谱范数 ||B_l||_2 以 1 为上界，因此残差变换是非扩张的，这在前向传播和反向传播中均增加了数值稳定性。此外，集合 M 在矩阵乘法下是封闭的，这保证了 mHC 深层堆叠场景下的稳定性。另外，输入变换 A_l 和输出变换 C_l 也通过 Sigmoid 函数约束为非负有界，以避免信号抵消的风险。

动态参数化。三个线性映射的参数是动态生成的，被分解为动态（输入依赖）分量和静态（输入无关）分量。给定输入 X_l ∈ R^(n_hc × d)，首先将其展平并归一化：X̂_l = RMSNorm(vec(X_l)) ∈ R^(1 × n_hc d)。然后，我们遵循传统 HC 生成无约束的原始参数 Ã_l ∈ R^(1 × n_hc)、B̃_l ∈ R^(n_hc × n_hc) 和 C̃_l ∈ R^(n_hc × 1)：

Ã_l = α_l^pre · (X̂_l W_l^{pre}) + S_l^pre,   (3)
B̃_l = α_l^res · Mat(X̂_l W_l^res) + S_l^res,   (4)
C̃_l = α_l^post · (X̂_l W_l^post) + S_l^post,   (5)

其中 W_l^pre, W_l^post ∈ R^(n_hc d × n_hc) 和 W_l^res ∈ R^(n_hc d × n_hc^2) 是用于生成动态分量的可学习参数；Mat(·) 将大小为 1 × n_hc^2 的向量重塑为大小为 n_hc × n_hc 的矩阵；S_l^pre ∈ R^(1 × n_hc)、S_l^post ∈ R^(n_hc × 1) 和 S_l^res ∈ R^(n_hc × n_hc) 是可学习的静态偏置；α_l^pre、α_l^res、α_l^post 是初始化为较小值的可学习门控因子。

施加参数约束。获得无约束的原始参数 Ã_l、B̃_l、C̃_l 后，我们对它们施加前述约束以增强数值稳定性。具体而言，对于输入和输出映射，我们采用 Sigmoid 函数 σ(·) 确保其非负性和有界性：

A_l = σ(Ã_l),   (6)
C_l = 2σ(C̃_l).   (7)

至于残差映射 B̃_l，我们将其投影到双随机矩阵流形 M 上。这通过 Sinkhorn-Knopp 算法实现，该算法首先对 B̃_l 施加指数函数以确保正定性，得到 M^(0) = exp(B̃_l)，然后迭代执行列归一化和行归一化：

M^(t) = T_r(T_c(M^(t-1))),   (8)

其中 T_r 和 T_c 分别表示行归一化和列归一化。此迭代收敛到一个受约束的双随机矩阵 B_l = M^(t_max)。我们选取 t_max = 20 作为实际值。

### 2.3 基于 CSA 和 HCA 的混合注意力

当上下文长度达到极端规模时，注意力机制成为模型中主要的计算瓶颈。对于 DeepSeek-V4，我们设计了两类高效注意力架构——压缩稀疏注意力（CSA）和重度压缩注意力（HCA）——并采用它们交替混合的配置，大幅降低了长文本场景中注意力的计算开销。CSA 集成了压缩和稀疏注意力策略：它首先将每 m 个 token 的 KV cache 压缩为一个条目，然后应用 DeepSeek 稀疏注意力（DSA）(DeepSeek-AI, 2025)，其中每个查询 token 仅关注 k 个压缩后的 KV 条目。HCA 追求极致压缩，将每 m'（≫ m）个 token 的 KV cache 合并为单个条目。CSA 和 HCA 的混合架构显著提升了 DeepSeek-V4 系列的长上下文效率，使百万 token 上下文在实践中成为可能。本小节描述混合注意力架构的核心技术，我们还提供了开源实现以无歧义地指定更多细节。

#### 2.3.1 压缩稀疏注意力

CSA 的核心架构如图 3 所示，它首先将每 m 个 token 的 KV cache 压缩为一个条目，然后应用 DeepSeek 稀疏注意力以进一步加速。

压缩键值条目。设 H ∈ R^(n × d) 为输入隐藏状态序列，其中 n 为序列长度，d 为隐藏维度。CSA 首先计算两组 KV 条目 C^a, C^b ∈ R^(n × c) 及其对应的压缩权重 Z^a, Z^b ∈ R^(n × c)，其中 c 为头维度：

C^a = H · W^{aKV},   Z^a = H · W^{aZ},   (9)
C^b = H · W^{bKV},   Z^b = H · W^{bZ},   (10)

其中 W^{aKV}、W^{bKV}、W^{aZ}、W^{bZ} ∈ R^(d × c) 为可训练参数。接下来，C^a 和 C^b 中每 m 个 KV 条目将根据其压缩权重和可学习位置偏置 B^a, B^b ∈ R^(m × c) 被压缩为一个条目，生成 C^Comp ∈ R^(n/m × c)。每个压缩条目 C_i^Comp ∈ R^c 的计算方式为

[S^a_{mi:m(i+1)-1}; S^b_{m(i-1):mi-1}] = Softmax_row([Z^a_{mi:m(i+1)-1} + B^a; Z^b_{m(i-1):mi-1} + B^b]),   (11)

C_i^Comp = Σ_{j=mi}^{m(i+1)-1} S^a_j ⊙ C^a_j + Σ_{j=m(i-1)}^{mi-1} S^b_j ⊙ C^b_j,   (12)

其中 ⊙ 表示 Hadamard product；Softmax_row(·) 表示沿行维度的 softmax 操作，对来自 Z^a 和 Z^b 的共 2m 个元素进行归一化。当 i = 0 时，Z^b_{m(i-1):mi-1} 以负无穷填充，C^b_{m(i-1):mi-1} 以零填充。注意，每个 C_i^Comp 由 2m 个 KV 条目推导而来，但用于 C_i^Comp 的 C^b 索引与用于 C_{i-1}^Comp 的 C^a 索引存在重叠。因此，CSA 实际上将序列长度压缩为原来的 1/m。

用于稀疏选择的 Lightning Indexer。获得压缩 KV 条目 C^Comp 后，CSA 应用 DSA 策略为核心注意力选择 top-k 个压缩 KV 条目。首先，CSA 对 C^Comp 执行相同的压缩操作，得到压缩索引器键 K^{IComp} ∈ R^(n/m × c_I)，其中 c_I 为索引器头维度。然后，对于查询 token t，我们以低秩方式生成索引器查询 {q^I_{t,1}; q^I_{t,2}; ...; q^I_{t,n_h^I}}：

c_t^Q = h_t · W^{DQ},   (13)
[q^I_{t,1}; q^I_{t,2}; ...; q^I_{t,n_h^I}] = q_t^I = c_t^Q · W^{IUQ},   (14)

其中 h_t ∈ R^d 为查询 token t 的输入隐藏状态；c_t^Q ∈ R^{d_c} 为查询的压缩潜向量；d_c 表示查询压缩维度；n_h^I 表示索引器查询头数；W^{DQ} ∈ R^(d × d_c) 和 W^{IUQ} ∈ R^(d_c × c_I n_h^I) 分别为索引器查询的降投影和升投影矩阵。接下来，查询 token t 与前序压缩块 s（s < Floor(m/t)）之间的索引分数 I_{t,s} ∈ R 计算如下

[w^I_{t,1}; w^I_{t,2}; ...; w^I_{t,n_h^I}] = w_t^I = h_t · W^w,   (15)

I_{t,s} = Σ_{h=1}^{n_h^I} (w^I_{t,h} · ReLU(q^I_{t,h} · K_s^{IComp})),   (16)

其中 W^w ∈ R^(d × n_h^I) 为可学习矩阵；w^I_{t,h} ∈ R 为第 h 个索引器头的权重。对于查询 token t，给定其索引分数 I_{t,:}，我们采用 top-k 选择器选择性地保留一部分压缩 KV 条目 C_t^{SprsComp} 用于后续核心注意力：

C_t^{SprsComp} = {C_s^Comp | I_{t,s} ∈ Top-k(I_{t,:})}.   (17)

共享键值多查询注意力。选择稀疏 KV 条目后，CSA 以多查询注意力（MQA）(Shazeer, 2019) 方式执行核心注意力，其中 C_t^{SprsComp} 中的每个压缩 KV 条目同时充当注意力键和值。具体而言，对于查询 token t，我们首先从压缩潜向量 c_t^Q 生成注意力查询 {q_{t,1}; q_{t,2}; ...; q_{t,n_h}}：

[q_{t,1}; q_{t,2}; ...; q_{t,n_h}] = q_t = c_t^Q · W^{UQ},   (18)

其中 n_h 表示查询头数；W^{UQ} ∈ R^(d_c × c n_h) 为查询的升投影矩阵。注意，潜查询向量 c_t^Q 与索引器查询所使用的共享。接下来，我们在 {q_{t,i}} 和 C_t^{SprsComp} 上执行 MQA：

o_{t,i} = CoreAttn(query=q_{t,i}, key=C_t^{SprsComp}, value=C_t^{SprsComp}),   (19)

其中 o_{t,i} ∈ R^c 为第 t 个 token 第 i 个头的核心注意力输出；CoreAttn(·) 表示核心注意力操作。

分组输出投影。在 DeepSeek-V4 的配置中，c·n_h 相当大。因此，直接将核心注意力操作的输出 [o_{t,1}; o_{t,2}; ...; o_{t,n_h}] = o_t ∈ R^(c n_h) 投影到 d 维隐藏状态将带来巨大的计算负担。为降低此开销，我们设计了分组输出投影策略。具体而言，我们首先将 n_h 个输出分为 g 组，然后对于每组输出 o_{t,i}^G ∈ R^(c n_h/g)，将其投影到 d_g 维的中间输出 o'_{t,i}^G ∈ R^{d_g}，其中 d_g < c n_h/g。最后，我们将中间输出 [o'_{t,1}^G; o'_{t,2}^G; ...; o'_{t,g}^G] ∈ R^(d_g g) 投影为最终的注意力输出 ô_t ∈ R^d。

#### 2.3.2 重度压缩注意力

HCA 的核心架构如图 4 所示，它以更重的方式压缩 KV cache，但不采用稀疏注意力。

压缩键值条目。总体而言，HCA 的压缩策略与 CSA 类似，但采用更大的压缩率 m'（≫ m）且不执行重叠压缩。设 H ∈ R^(n × d) 为输入隐藏状态序列，HCA 首先计算原始 KV 条目 C ∈ R^(n × c) 及其对应的压缩权重 Z ∈ R^(n × c)：

C = H · W^{KV},   (20)
Z = H · W^Z,   (21)

其中 W^{KV}、W^Z ∈ R^(d × c) 为可训练参数。接下来，C 中每 m' 个 KV 条目将根据压缩权重和可学习位置偏置 B ∈ R^(m' × c) 被压缩为一个条目，生成 C^Comp ∈ R^(n/m' × c)。每个压缩条目 C_i^Comp ∈ R^c 的计算方式为

S_{m'i:m'(i+1)-1} = Softmax_row(Z_{m'i:m'(i+1)-1} + B),   (22)
C_i^Comp = Σ_{j=m'i}^{m'(i+1)-1} S_j ⊙ C_j.   (23)

通过此压缩操作，HCA 将序列长度压缩为原来的 1/m'。

共享键值 MQA 和分组输出投影。HCA 同样采用共享 KV MQA 和分组输出投影策略。KV 压缩后，对于查询 token t，HCA 首先以低秩方式生成注意力查询 {q_{t,1}; q_{t,2}; ...; q_{t,n_h}}：

c_t^Q = h_t · W^{DQ},   (24)
[q_{t,1}; q_{t,2}; ...; q_{t,n_h}] = q_t = c_t^Q · W^{UQ},   (25)

其中 h_t ∈ R^d 为查询 token t 的输入隐藏状态；n_h 表示查询头数；W^{DQ} ∈ R^(d × d_c) 和 W^{UQ} ∈ R^(d_c × c n_h) 分别为查询的降投影和升投影矩阵。接下来，我们在 {q_{t,i}} 和 C^Comp 上执行 MQA：

o_{t,i} = CoreAttn(query=q_{t,i}, key=C^Comp, value=C^Comp),   (26)

其中 o_{t,i} ∈ R^c 为第 t 个 token 第 i 个头的核心注意力输出。接下来，与 CSA 一样，HCA 将 n_h 个输出分为 g 组，对于每组输出 o_{t,i}^G ∈ R^(c n_h/g)，HCA 将其投影到 d_g 维的中间输出 o'_{t,i}^G ∈ R^{d_g}，其中 d_g < c n_h/g。最后，HCA 将中间输出 [o'_{t,1}^G; o'_{t,2}^G; ...; o'_{t,g}^G] ∈ R^(d_g g) 投影为最终的注意力输出 ô_t ∈ R^d。

#### 2.3.3 其他细节

除了上述 CSA 和 HCA 的核心架构外，我们的混合注意力还包含若干其他技术。为行文简洁，我们在上述介绍中省略了这些附加技术，并将在本小节中简要描述。此外，本小节仅关注这些技术的核心思想，为简洁起见可能省略一些微小细节。我们鼓励读者参考我们的开源实现以获取无歧义的细节。

查询和键值条目归一化。对于 CSA 和 HCA，我们在核心注意力操作之前，对查询的每个头和压缩 KV 条目的唯一头执行额外的 RMSNorm 操作。此归一化避免了注意力 logits 爆炸，并可能提升训练稳定性。

部分旋转位置编码。对于 CSA 和 HCA，我们对注意力查询、KV 条目和核心注意力输出部分采用旋转位置编码（RoPE）(Su et al., 2024)。具体而言，对于 CSA 和 HCA 中使用的每个查询向量和 KV 条目向量，我们对其最后 64 维应用 RoPE。由于 KV 条目同时充当注意力键和值，朴素的核心注意力输出 {o_{t,i}} 将携带从 KV 条目加权和推导而来的绝对位置编码。作为对策，我们同样对每个 o_{t,i} 的最后 64 维施加位置为 -i 的 RoPE。这样，核心注意力的输出也将携带相对位置编码——每个 KV 条目对核心注意力输出的贡献也将与查询和 KV 条目之间的距离相关。

滑动窗口注意力附加分支。为了严格保持 CSA 和 HCA 中的因果性，每个查询仅关注前序压缩 KV 块。因此，查询无法访问同一压缩块内其他 token 的信息。同时，在语言建模中，近邻 token 通常与查询 token 具有更大的相关性。基于这些原因，我们以滑动窗口方式为 CSA 和 HCA 引入了一个补充注意力分支，以更好地建模局部依赖关系。具体而言，对于每个查询 token，我们额外生成对应最近 n_win 个 token 的 n_win 个未压缩 KV 条目。在 CSA 和 HCA 的核心注意力中，滑动窗口中的这些 KV 条目将与压缩 KV 条目一起使用。

注意力汇。在 CSA 和 HCA 的核心注意力中，我们采用了注意力汇（attention sink）技巧 (OpenAI, 2025; Xiao et al., 2024)。具体而言，我们设置一系列可学习的汇 logits {z'_1, z'_2, ..., z'_{n_h}}。对于第 h 个注意力头，Exp(z'_h) 将被加到注意力分数的分母上：

s_{h,i,j} = Exp(z_{h,i,j}) / (Σ_k Exp(z_{h,i,k}) + Exp(z'_h)),   (27)

其中 s_{h,i,j}、z_{h,i,j} ∈ R 分别表示第 h 个注意力头在第 i 个查询 token 与第 j 个前序 token 或压缩块之间的注意力分数和注意力 logit。此技术允许每个查询头调整其总注意力分数使其不等于 1，甚至可以接近 0。

#### 2.3.4 效率讨论

由于采用了混合 CSA 和 HCA，以及低精度计算和存储，DeepSeek-V4 系列的注意力模块在注意力 FLOPs 和 KV cache 大小方面均实现了显著效率，尤其在长上下文场景中。首先，我们对 KV 条目采用混合存储格式：旋转位置编码（RoPE）维度使用 BF16 精度，其余维度使用 FP8 精度。这种混合表示相比纯 BF16 存储将近乎减半 KV cache 大小。其次，Lightning Indexer 内的注意力计算以 FP4 精度执行，加速了极长上下文下的注意力操作。第三，相对于 DeepSeek-V3.2，DeepSeek-V4 系列选择了更小的注意力 top-k，从而提升了模型在短中长度文本上的效率。最后，也是最重要的，压缩注意力和混合注意力技术大幅降低了 KV cache 大小和计算 FLOPs。

以头维度为 128 的 BF16 GQA8 (Ainslie et al., 2023) 为基线——这是 LLM 注意力的常见配置之一——在 1M 上下文设置下，DeepSeek-V4 系列的 KV cache 大小可大幅缩减至该基线的约 2%。

此外，即使与 DeepSeek-V3.2 (DeepSeek-AI, 2025)——本身已是一个高效基线——相比，DeepSeek-V4 系列在效率方面仍展现出显著优势。图 1 右侧部分提供了两者推理 FLOPs 和 KV cache 大小的比较。

### 2.4 Muon 优化器

我们采用 Muon (Jordan et al., 2024; Liu et al., 2025) 优化器用于 DeepSeek-V4 系列中的大多数模块，因为它具有更快的收敛速度和更高的训练稳定性。我们的 Muon 优化化完整算法总结在算法 1 中。

算法 1 DeepSeek-V4 的 Muon 优化器
输入：学习率 η，动量 μ，权重衰减 λ，更新重缩放因子 γ
1: for 每个训练步骤 t do
2:   for 每个逻辑独立的权重 W ∈ R^(n × m) do
3:     G_t = ∇_W L_t(W_{t-1})            ▹ 计算梯度
4:     M_t = μM_{t-1} + G_t               ▹ 累积动量缓冲
5:     O'_t = HybridNewtonSchulz(μM_t + G_t)  ▹ Nesterov 技巧和混合 Newton-Schulz
6:     O_t = O'_t · max(n, m) · γ          ▹ 重缩放更新的 RMS
7:     W_t = W_{t-1} · (1 - ηλ) - ηO_t    ▹ 执行权重衰减和更新
8:   end for
9: end for

基本配置。我们对嵌入模块、预测头模块、mHC 模块的静态偏置和门控因子，以及所有 RMSNorm 模块的权重保持使用 AdamW (Loshchilov and Hutter, 2017) 优化器。所有其他模块使用 Muon 更新。遵循 Liu et al. (2025)，我们同样对 Muon 参数施加权重衰减，使用 Nesterov (Jordan et al., 2024; Nesterov, 1983) 技巧，并重缩放更新矩阵的均方根（RMS）以复用 AdamW 超参数。与他们的方法不同，我们使用混合 Newton-Schulz 迭代进行正交化。

混合 Newton-Schulz 迭代。对于给定矩阵 M，设其奇异值分解（SVD）为 M = UΣV^T。Newton-Schulz 迭代旨在将 M 近似正交化为 UV^T。通常，M 首先被归一化为 M_0 = M/||M||_F 以确保其最大奇异值不超过 1。然后，每次 Newton-Schulz 迭代执行以下操作：

M_k = aM_{k-1} + b(M_{k-1}M_{k-1}^T)M_{k-1} + c(M_{k-1}M_{k-1}^T)^2 M_{k-1}.   (28)

我们的混合 Newton-Schulz 在两个不同阶段执行 10 次迭代。在前 8 步中，我们使用系数 (a, b, c) = (3.4445, -4.7750, 2.0315) 以驱动快速收敛，使奇异值接近 1。在最后 2 步中，我们切换到系数 (a, b, c) = (2, -1.5, 0.5)，将奇异值精确稳定在 1。

避免注意力 logits 爆炸。DeepSeek-V4 系列的注意力架构允许我们直接对注意力查询和 KV 条目施加 RMSNorm，有效防止注意力 logits 爆炸。因此，我们在 Muon 优化器中不采用 QK-Clip 技术 (Liu et al., 2025)。

## 3 通用基础设施

### 3.1 专家并行中细粒度通信-计算重叠

混合专家（MoE）可通过专家并行（EP）加速。然而，EP 需要复杂的多节点通信，对互连带宽和延迟提出了很高的要求。为缓解 EP 中的通信瓶颈，并在更低的互连带宽需求下实现更高的端到端性能，我们提出了一种细粒度 EP 方案，将通信和计算融合为单一流水线内核，实现通信-计算重叠。

通信延迟可以被隐藏。我们 EP 方案的关键洞察是，MoE 层中的通信延迟可以被计算有效隐藏。如图 5 所示，在 DeepSeek-V4 系列中，每个 MoE 层主要可分解为四个阶段：两个通信受限阶段——Dispatch 和 Combine，以及两个计算受限阶段——Linear-1 和 Linear-2。我们的性能分析表明，在单个 MoE 层内，通信的总时间少于计算的总时间。因此，将通信和计算融合为统一流水线后，计算仍然是主要瓶颈，这意味着系统可以容忍更低的互连带宽而不降低端到端性能。

细粒度 EP 方案。为进一步降低互连带宽需求并放大重叠的收益，我们引入了更细粒度的专家划分方案。受多项相关工作（Aimuyo et al., 2025; Zhang et al., 2025b）的启发，我们将专家拆分并调度为多个波次（wave）。每个波次包含一小部分专家。一旦波次内的所有专家完成通信，计算可以立即开始而无需等待其他专家。在稳态下，当前波次的计算、下一波次的 token 传输以及已完成专家的结果发送全部并发进行，如图 5 所示。这在专家之间形成了细粒度流水线，使计算和通信在整个波次期间保持连续。基于波次的调度加速了极端场景下的性能，例如强化学习（RL）rollout，通常会遇到长尾小批量问题。

性能与开源 Mega-Kernel。我们在 NVIDIA GPU 和华为昇腾 NPU 平台上验证了细粒度 EP 方案。与强大的非融合基线相比，该方案在通用推理工作负载上实现了 1.50 ∼ 1.73× 的加速，在 RL rollout 和高速智能体服务等延迟敏感场景中加速比最高达 1.96×。我们已开源基于 CUDA 的 mega-kernel 实现，命名为 MegaMoE，作为 DeepGEMM 的组件。

观察与建议。我们分享了内核开发中的观察和经验教训，并向硬件厂商提出一些建议，以期助力高效硬件设计并实现更好的软硬件协同设计：

• 计算-通信比。完整的通信-计算重叠取决于计算-通信比，而非仅取决于带宽。设峰值计算吞吐量为 C，互连带宽为 B，当 C/B ≤ V_comp/V_comm 时，通信可以被完全隐藏，其中 V_comp 表示计算量，V_comm 表示通信量。对于 DeepSeek-V4-Pro，每个 token-expert 对需要 6hd FLOPs（SwiGLU 的 gate、up 和 down 投影），但仅需 3h 字节的通信（FP8 Dispatch + BF16 Combine），这简化为：C/B ≤ 2d = 6144 FLOPs/Byte。也就是说，每 GBps 的互连带宽足以隐藏 6.1 TFLOP/s 计算的通信。一旦带宽满足此阈值，它就不再是瓶颈，将额外的硅片面积用于进一步提升带宽带来的收益递减。我们鼓励未来的硬件设计瞄准此类平衡点，而非无条件地扩展带宽。

• 功耗预算。极端的内核融合驱动计算、内存和网络同时达到高负载，使功耗节流成为关键性能限制因素。我们建议未来的硬件设计为此类完全并发的工作负载提供充足的功耗裕量。

• 通信原语。在 dispatch 阶段，我们采用拉取式（pull-based）方法，即每个 GPU 主动从远程 GPU 读取激活值，避免了细粒度推送（push）带来的高通知延迟。未来具有更低延迟跨 GPU 信令的硬件将使推送变得可行，并支持更自然的通信模式。

• 激活函数。我们建议用低成本的逐元素激活函数替换 SwiGLU，该函数不涉及指数或除法运算。这直接减少了 GEMM 后处理的开销，防止 GEMM 流水线因激活函数计算而停滞，从而提升整体计算吞吐量和资源利用率。

### 3.2 基于 TileLang 的灵活高效内核开发

在实践中，我们精细的模型架构将产生数百个细粒度的 Torch ATen 算子。我们采用 TileLang（Wang et al., 2026）开发一组融合内核来替代其中的绝大多数，以最小的代价实现最优性能。它还允许我们在验证期间快速原型化注意力变体等算子。这些内核在模型架构开发、大规模训练以及最终的推理服务生产部署中发挥着关键作用。作为一种领域特定语言（DSL），TileLang 在开发效率和运行时效率之间取得平衡，支持快速开发的同时允许在同一代码库中进行深度的迭代优化。此外，我们与 TileLang 社区密切合作，以促进更敏捷、高效和稳定的内核开发工作流。

通过 Host Codegen 降低调用开销。随着加速器性能持续增长，CPU 端编排开销变得日益突出。对于小型、高度优化的内核，此类固定宿主开销很容易成为利用率和吞吐量的上限。这种开销的一个常见来源是宿主端逻辑（如运行时契约检查）通常以 Python 编写以保持灵活性，因此每次调用都会产生固定开销。我们通过 Host Codegen 缓解此开销，将大部分宿主端逻辑移入生成的宿主代码中。具体而言，我们首先在 IR（中间表示）层面共同生成设备内核和轻量级宿主启动器，嵌入从语言前端解析的必要元数据——如数据类型、秩/形状约束和步幅/布局假设。然后，启动器被降低为基于 TVM-FFI（Chen et al., 2018）框架构建的宿主源代码，其紧凑的调用约定和零拷贝张量互操作共同最小化了宿主端开销。在运行时，此生成的宿主代码执行验证和参数编组，将所有每次调用的检查移出 Python 执行路径。我们的测量表明，CPU 端验证开销从每次调用的数十或数百微秒降至不到一微秒。

SMT 求解器辅助的形式化整数分析。TileLang 内核涉及复杂的张量索引算术，需要强有力的形式化整数分析。在布局推断、内存危害检测和边界分析等编译遍历中，编译器必须验证整数表达式是否满足特定属性以启用相应的优化。因此，更强的形式化分析能力可以解锁更高级和更复杂的优化机会。为此，我们将 Z3 SMT 求解器（De Moura and Bjørner, 2008）集成到 TileLang 的代数系统中，为张量程序中的大多数整数表达式提供形式化分析能力。我们通过将 TileLang 的整数表达式转换为 Z3 的无量词非线性整数算术（QF_NIA），在计算开销和形式化表达能力之间取得平衡。基于整数线性规划（ILP）求解器，QF_NIA 无缝解决了内核中常见的标准线性整数表达式。此外，其固有的非线性推理能力有效解决了可变张量形状上向量化等高级挑战。在合理的资源限制下，Z3 在将编译时间开销限制在仅几秒的同时提升了整体优化性能。其影响在多个编译遍历中都很显著，包括向量化、屏障插入和代码简化。

数值精度与逐位可复现性。在生产环境中，数值正确性和可复现性与原始吞吐量同等重要。因此我们默认优先保证精度：编译器级别禁用快速数学优化，影响精度的近似仅作为显式选入的前端算子提供（例如 T.__exp、T.__log 和 T.__sin）。反之，当需要严格的 IEEE-754 语义时，TileLang 提供具有显式舍入模式的 IEEE 兼容内建函数（例如 T.ieee_fsqrt、T.ieee_fdiv 和 T.ieee_add），使开发者能够精确指定数值行为。我们还以逐位可复现性为目标，以对照手写 CUDA 基线验证内核。我们将 TileLang 的代数简化和降低规则与主流 CUDA 工具链（如 NVCC）对齐，以避免引入非预期的比特级差异的变换。布局注解（例如 T.annotate_layout）进一步允许用户固定依赖布局的降低决策，使求值和累加顺序与参考 CUDA 实现保持一致，从而在需要时实现比特级相同的输出。我们的评估表明，这些面向精度和可复现性的设计选择不会牺牲性能：在保守默认设置下，TileLang 内核仍具竞争力，同时暴露旋钮以选择性地放宽数值约束以获取更高速度。

### 3.3 高性能批量不变与确定性内核库

为实现高效训练和推理，我们开发了一套全面的高性能计算内核。除基本功能和最大化硬件利用率外，另一个关键设计目标是确保训练可复现性以及预训练、后训练和推理流水线之间的逐位对齐。因此，我们以最小的性能开销实现了端到端的、逐位批量不变和确定性内核。这些内核有助于调试、稳定性分析和一致的后训练行为。

批量不变性。批量不变性确保任意给定 token 的输出保持逐位相同，无论其在批量中的位置如何。为实现批量不变性，主要挑战如下：

• 注意力。为实现批量不变性，我们不能使用 split-KV 方法（Dao et al., 2023），该方法将单个序列的注意力计算分布到多个流多处理器（SM）上以平衡 SM 的负载。然而，放弃此技术将导致严重的波量化（wave-quantization）问题，可能对 GPU 利用率产生不利影响。为此，我们为批量不变解码开发了双内核策略。第一个内核在单个 SM 内计算整个序列的注意力输出，确保完全占满波次的高吞吐量。第二个内核为最小化最后部分填充波次的延迟从而缓解波量化，使用多个 SM 处理单个序列。为确保这两个内核的逐位一致性，我们精心设计了第二个内核的计算路径，确保其累加顺序与第一个内核相同。此外，第二个内核利用线程块簇内的分布式共享内存，实现跨 SM 的高速数据交换。这种双内核方法有效地将批量不变解码的开销限制在可忽略范围内。

• 矩阵乘法。传统的 cuBLAS 库（NVIDIA Corporation, 2024）无法实现批量不变性。因此，我们端到端地用 DeepGEMM（Zhao et al., 2025）替换了它。此外，对于极小的批量大小，常规实现通常采用 split-k（Osama et al., 2023）技术来提升性能。遗憾的是，split-k 技术无法保证批量不变性，而这是 DeepSeek-V4 中的关键特性。因此，我们在大多数场景中放弃了 split-k，但这可能导致性能下降。为解决此问题，我们引入了一系列优化，使我们的矩阵乘法实现在大多数主要场景中能够匹配甚至超越标准 split-k 的性能。

确定性。确定性训练对于调试硬件或软件问题极为有益。此外，当训练出现损失尖峰等异常时，确定性使研究人员能够更容易地定位数值原因并进一步完善模型设计。训练中的非确定性通常源于非确定性的累加顺序，往往由于使用原子加法指令所致。此问题主要发生在反向传播中，尤其在以下部分：

• 注意力反向传播。在稀疏注意力的反向传播常规实现中，我们使用 atomicAdd 累加 KV token 的梯度。由于浮点加法的非结合性，这引入了非确定性。为解决此问题，我们为每个 SM 分配独立的累加缓冲区，然后对所有缓冲区执行全局确定性求和。

• MoE 反向传播。当来自不同 rank 的多个 SM 同时向接收 rank 上的同一缓冲区写入数据时，协商写入位置也会引入非确定性。为解决此问题，我们在每个 rank 内设计了 token 顺序预处理机制，结合跨多个 rank 的缓冲区隔离。此策略确保了专家并行发送结果和 MoE 反向传播中累加顺序的确定性。

• mHC 中的矩阵乘法。mHC 涉及输出维度仅为 24 的矩阵乘法。对于极小的批量大小，我们被迫使用 split-k（Osama et al., 2023）算法，其朴素实现将导致非确定性。为克服此问题，我们分别输出每个 split 部分，然后在后续内核中执行确定性归约，从而同时保持性能和确定性。

### 3.4 训练框架

我们的训练框架建立在为 DeepSeek-V3（DeepSeek-AI, 2024）开发的可扩展高效基础设施之上。在训练 DeepSeek-V4 时，我们继承了这一稳健的基础，同时引入了若干关键创新以适应其新颖的架构组件——特别是 Muon 优化器、mHC 和混合注意力机制——同时保持高训练效率和稳定性。

#### 3.4.1 Muon 的高效实现

Muon 优化器需要完整的梯度矩阵来计算参数更新，这在与零冗余优化器（ZeRO）（Rajbhandari et al., 2020）结合时带来了挑战。传统 ZeRO 为 AdamW 等逐元素优化器设计，其中单个参数矩阵可以跨多个 rank 分区和更新。为解决此冲突，我们为 Muon 设计了 ZeRO 桶分配的混合策略。对于稠密参数，我们限制 ZeRO 并行的最大规模，并采用背包算法将参数矩阵分配到这些 rank 上，确保每个 rank 管理大致均衡的负载。每个 rank 上的桶被填充至与所有 rank 中最大桶相同的大小，以促进高效的 reduce-scatter 操作。在我们的设置中，这种填充通常带来不到 10% 的内存开销，其中每个 rank 管理不超过五个参数矩阵。当数据并行的总规模超过 ZeRO 的限制时，我们在额外的数据并行组中冗余计算 Muon 更新，以计算换取减少的总桶内存。对于 MoE 参数，我们独立优化每个专家。我们首先将所有层所有专家的 SwiGLU（Shazeer, 2020）中所有 down 投影矩阵展平，随后是展平的 up 投影矩阵和 gate 矩阵。然后，我们对展平向量进行填充，确保可以将此向量均匀分配到所有 rank 上而不拆分任何逻辑独立的矩阵。鉴于专家数量众多，我们对 MoE 参数不施加 ZeRO 并行限制，填充开销也可忽略。此外，在每个 rank 上，相同形状的连续参数将被自动合并，实现 Newton-Schulz 迭代的批量执行以提高硬件利用率。此外，我们观察到 Muon 中的 Newton-Schulz 迭代在使用 BF16 矩阵乘法计算时保持稳定。利用这一点，我们进一步以随机舍入方式将跨数据并行 rank 同步的 MoE 梯度量化为 BF16 精度，将通信量减半。为避免低精度加法器引入的累加误差，我们用两阶段方法替代了传统的树形或环形 reduce-scatter 集合通信。首先，all-to-all 操作在 rank 之间交换本地梯度，然后每个 rank 在 FP32 中执行本地求和。此设计保持了数值鲁棒性。

#### 3.4.2 mHC 的高性价比与内存高效实现

与传统残差连接相比，mHC 的引入增加了激活内存消耗和流水线阶段之间的通信量。为缓解这些开销，我们实施了几项优化策略。首先，我们精心设计并实现了 mHC 在训练和推理中的融合内核。其次，我们引入了选择性检查点中间张量的重计算策略。具体而言，我们重计算层间的大部分隐藏状态和所有归一化层输入，同时避免重计算计算密集型操作。这在内存节省和计算开销之间实现了平衡。第三，我们调整了 DualPipe 1F1B 重叠方案以适应增加的流水线通信，并实现 mHC 中某些操作的并发执行。综合来看，这些优化将 mHC 的挂钟时间开销限制在重叠 1F1B 流水线阶段的仅 6.7%。更多工程优化的细节可在 mHC 专题论文（Xie et al., 2026）中找到。

#### 3.4.3 长上下文注意力的上下文并行

传统上下文并行（CP）沿序列维度划分，每个 rank 维护连续的 s 个 token。这对我们的压缩注意力机制（即 CSA 和 HCA）引入了两个挑战。一方面，训练样本从多个序列打包而来，每个序列独立地以因子 m（或 m'）压缩，不足 m 个的尾部 token 被丢弃。因此，压缩后的 KV 长度通常小于 s/m，且各 rank 之间不同。另一方面，压缩需要 m 个连续的 KV 条目，这些条目可能跨越两个相邻 CP rank 之间的边界。为解决这些挑战，我们设计了两阶段通信方法。在第一阶段，每个 rank i 将其最后 m 个未压缩 KV 条目发送给 rank i + 1。然后，rank i + 1 将接收到的部分条目与其本地 s 个未压缩 KV 条目一起压缩，产生固定长度为 s/m + 1 的压缩条目，其中存在一些填充条目。在第二阶段，跨所有 CP rank 的 all-gather 操作收集本地压缩 KV 条目。然后，融合的 select-and-pad 算子将它们重组为完整的压缩 KV 条目集合，总长度为 cp_size · s/m。所有填充条目放置在尾部。对于 HCA 和 CSA 中的索引器，每个查询 token 可见的压缩 KV 条目范围可以通过规则预计算。对于 CSA 中的稀疏注意力，top-k 选择器显式指定每个查询可见的压缩 KV 条目索引。

#### 3.4.4 面向灵活激活检查点的扩展自动微分

传统激活检查点实现以整个模块为粒度操作，决定在反向传播期间是保留还是重计算其输出激活。这种粗粒度通常导致重计算代价和激活内存占用之间的次优权衡。另一种方法是手动实现整层的前向和反向逻辑，显式管理张量检查点状态。虽然这实现了细粒度控制，但该方法失去了自动微分框架的便利性，大幅增加了开发复杂度。为实现细粒度控制而不牺牲编程效率，我们实现了具有自动微分支持的张量级激活检查点机制。借助此机制，开发者只需实现前向传播并选择性地标注个别张量以进行自动检查点和重计算。我们的框架利用 TorchFX（Reed et al., 2022）追踪完整的计算图。对于每个被标注的张量，它执行反向遍历以识别其重计算所需的最小子图。我们将这些最小子图定义为重计算图，并将其插入到相应梯度计算之前的反向逻辑中。与手动实现相比，此设计在训练期间不引入额外开销。此框架中的重计算通过直接释放被标注张量的 GPU 内存并复用重计算张量的存储指针来实现，无需任何 GPU 内存拷贝。此外，由于图追踪具体执行模型，我们可以跟踪每个张量的底层存储指针，从而实现对共享存储的张量（例如 reshape 操作的输入和输出）重计算的自动去重。这使开发者无需在标注重计算时推理底层内存细节。

### 3.5 推理框架

我们的推理框架在很大程度上继承自 DeepSeek-V3，在 KV Cache 管理方面有所不同。

#### 3.5.1 KV Cache 结构与管理

为高效管理 DeepSeek-V4 混合注意力机制产生的异构 KV cache，我们设计了定制的 KV cache 布局。该布局如图 6 所示，下面将详细阐述。

DeepSeek-V4 中的异构 KV 条目。DeepSeek-V4 系列的混合注意力机制引入了多种类型的 KV 条目，具有不同的键值（KV）cache 大小和更新规则。用于稀疏选择的 lightning indexer 向 KV cache 中引入了额外维度，这些维度具有与主注意力不同的嵌入大小。CSA 和 HCA 中采用的压缩技术分别以 1/m 和 1/m' 的因子缩减序列长度，从而降低了整体 KV cache 大小。因此，不同层的 KV cache 大小各不相同。此外，滑动窗口注意力（SWA）层也以不同的 KV cache 大小以及独立的缓存命中和驱逐策略运行。在压缩分支中，每 m 个 token 生成一个 KV 条目。当剩余 token 数不足以进行压缩时，所有待处理的 token 及其关联的隐藏状态必须保留在缓冲区中，直到可以执行压缩操作。这些缓冲的 token 代表由位置上下文确定的序列状态，也在 KV cache 框架内管理。

混合注意力 KV Cache 管理的挑战。混合注意力机制违反了 PagedAttention 及其变体的基本假设。尽管近期的混合 KV cache 管理算法（如 Jenga（Zhang et al., 2025a）、Hymba（Dong et al., 2025））针对通用混合注意力模型或特定结构，但两个主要障碍阻止了在 PagedAttention 框架下整合所有层的 KV cache：

• 多样的缓存策略，例如滑动窗口注意力中使用的策略。
• 高性能注意力内核施加的约束，包括对齐要求。

为高效管理 DeepSeek-V4 的 KV cache，我们设计了相应策略来克服这两个挑战。

SWA 和未压缩尾部 token 的状态缓存。为解决第一个障碍，我们采用替代缓存管理机制。由于 SWA 旨在有限 KV cache 大小下提升性能，将其与压缩分支中未压缩的尾部 token 一起视为状态空间模型是合理的。相应的 KV cache 因此可以被视为仅取决于当前位置的序列特定状态。据此，我们预分配固定且有限大小的状态缓存池，并动态分配给每个序列。

稀疏注意力内核协同设计。关于第二个障碍，传统高性能注意力内核通常假设每个块有固定数量 B 的 token 以优化性能，对应 CSA 中的 B · m 个原始 token 和 HCA 中的 B · m' 个原始 token。通过采用高性能稀疏注意力内核，不同层可以在不降低性能的情况下适应每个块的可变 token 数。实现这一点需要协同设计 KV cache 布局和稀疏注意力内核。例如，填充块以对齐缓存行可以提升性能。因此，对于压缩率为 m 的 CSA 和压缩率为 m' 的 HCA，每个块的原始 token 数可以是 lcm(m, m') 的任意倍数，即两个压缩率的最小公倍数。

#### 3.5.2 磁盘 KV Cache 存储

在服务 DeepSeek-V4 时，我们利用磁盘 KV cache 存储机制来消除共享前缀请求的重复预填充。对于 CSA/HCA 中的压缩 KV 条目和滑动窗口注意力（SWA）中的未压缩 KV 条目，我们分别设计了存储管理方案。

对于 CSA 和 HCA，我们简单地将所有压缩 KV 条目存储到磁盘。当请求命中已存储的前缀时，我们读取并复用与该前缀对应的压缩 KV 条目，直到最后一个完整的压缩块。特别地，对于尾部不完整块中的前缀 token，我们仍需重新计算以恢复未压缩 KV 条目，因为 CSA 和 HCA 中的未压缩 KV 条目不被存储。

对于 SWA KV 条目，由于它们未被压缩且存在于每一层，其数据量约为压缩 CSA 和 HCA KV 条目的 8 倍。为高效处理这些大型 SWA KV 条目，我们提出并实现了三种不同的磁盘 SWA KV 条目管理策略，每种策略在存储开销和计算冗余之间提供不同的权衡：

• 完整 SWA 缓存。此策略存储所有 token 的完整 SWA KV 条目，确保计算零冗余。在此策略下，命中前缀的 SWA KV 条目可以通过仅读取该前缀内最后 n_win 个 token 的磁盘缓存来重建。尽管计算零冗余，此策略对现代基于 SSD 的存储系统效率低下——每次命中请求仅访问所存储 SWA KV cache 的一小部分，这导致不均衡的写密集访问模式。

• 周期性检查点。此策略每隔 p 个 token 检查点最近 n_win 个 token 的 SWA KV 条目，其中 p 为可调参数。对于命中前缀，我们加载最近的检查点状态，然后重计算剩余的尾部 token。通过调节 p，此策略实现了存储和计算之间按需权衡。

• 零 SWA 缓存。此策略不存储任何 SWA KV 条目。对于命中前缀，我们需要执行更多重计算以恢复 SWA KV 条目。具体而言，在每个注意力层中，每个 token 的 SWA KV 条目仅依赖于前一层的最近 n_win 个 token 的 SWA KV 条目。因此，利用已缓存的 CSA 和 HCA KV 条目，重计算最后 n_win · L 个 token 即足以恢复 L 层模型的最后 n_win 个 SWA KV 条目。

根据具体部署场景，我们选择最合适的策略以实现存储和计算之间的期望权衡。

## 4 预训练

### 4.1 数据构建

在 DeepSeek-V3 预训练数据的基础上，我们致力于构建更加多样化、更高质量的训练语料库，并具有更长的有效上下文。我们持续完善数据构建流水线。对于网络来源的数据，我们实施过滤策略以移除批量自动生成和模板化内容，从而缓解模型崩溃的风险 (Zhu et al., 2024)。数学和编程语料仍然是训练数据的核心组成部分，我们通过在中期训练阶段引入智能体数据进一步增强 DeepSeek-V4 系列的编程能力。对于多语言数据，我们为 DeepSeek-V4 构建了更大规模的语料库，提升其对不同文化长尾知识的捕获能力。对于 DeepSeek-V4，我们特别强调长文档数据的策展，优先选择科学论文、技术报告及其他反映独特学术价值的材料。综合以上所有，我们的预训练语料库包含超过 32T token，涵盖数学内容、代码、网页、长文档及其他高质量类别。

对于预训练数据，我们大体上遵循与 DeepSeek-V3 相同的预处理策略。在分词方面，在 DeepSeek-V3 分词器的基础上，我们引入了少量用于上下文构建的特殊 token，词汇量仍保持为 128K。我们同样继承了 DeepSeek-V3 的 token 拆分 (DeepSeek-AI, 2024) 和 Fill-in-Middle (FIM) (DeepSeek-AI, 2024) 策略。受 Ding et al. (2024) 启发，我们将来自不同来源的文档打包到合适的序列中，以最小化样本截断。与 DeepSeek-V3 不同的是，我们在预训练期间采用样本级注意力掩码。

### 4.2 预训练设置

#### 4.2.1 模型设置

**DeepSeek-V4-Flash。** 我们将 Transformer 层数设置为 43，隐藏维度 d 设置为 4096。对于前两层，我们使用纯滑动窗口注意力。对于后续层，CSA 和 HCA 以交替方式使用。对于 CSA，我们设置压缩率 m 为 4，索引器查询头数 n_h^I 为 64，索引器头维度 c_I 为 128，稀疏注意力选择的 KV 条目数（即 attention top-k）为 512。对于 HCA，我们设置压缩率 m' 为 128。对于 CSA 和 HCA，我们设置查询头数 n_h 为 64，头维度 c 为 512，查询压缩维度 d_c 为 1024。输出投影组数 g 设置为 8，每个中间注意力输出维度 d_g 设置为 1024。对于滑动窗口注意力的附加分支，窗口大小 n_win 设置为 128。我们在所有 Transformer 块中采用 MoE 层，但对前 3 个 MoE 层使用 Hash routing 策略。每个 MoE 层由 1 个共享专家和 256 个路由专家组成，每个专家的中间隐藏维度为 2048。在路由专家中，每个 token 激活 6 个专家。多 token 预测深度设置为 1。对于 mHC，扩展因子 n_hc 设置为 4，Sinkhorn-Knopp 迭代次数 t_max 设置为 20。在此配置下，DeepSeek-V4-Flash 包含 284B 总参数，其中每个 token 激活 13B。

**DeepSeek-V4-Pro。** 我们将 Transformer 层数设置为 61，隐藏维度 d 设置为 7168。对于前两层，我们使用 HCA。对于后续层，CSA 和 HCA 以交替方式使用。对于 CSA，我们设置压缩率 m 为 4，索引器查询头数 n_h^I 为 64，索引器头维度 c_I 为 128，稀疏注意力选择的 KV 条目数（即 attention top-k）为 1024。对于 HCA，我们设置压缩率 m' 为 128。对于 CSA 和 HCA，我们设置查询头数 n_h 为 128，头维度 c 为 512，查询压缩维度 d_c 为 1536。输出投影组数 g 设置为 16，每个中间注意力输出维度 d_g 设置为 1024。对于滑动窗口注意力的附加分支，窗口大小 n_win 设置为 128。我们在所有 Transformer 块中采用 MoE 层，但对前 3 个 MoE 层使用 Hash routing 策略。每个 MoE 层由 1 个共享专家和 384 个路由专家组成，每个专家的中间隐藏维度为 3072。在路由专家中，每个 token 激活 6 个专家。多 token 预测深度设置为 1。对于 mHC，扩展因子 n_hc 设置为 4，Sinkhorn-Knopp 迭代次数 t_max 设置为 20。在此配置下，DeepSeek-V4-Pro 包含 1.6T 总参数，其中每个 token 激活 49B。

#### 4.2.2 训练设置

**DeepSeek-V4-Flash。** 我们对大多数参数采用 Muon 优化器 (Jordan et al., 2024; Liu et al., 2025)，但对嵌入模块、预测头模块以及所有 RMSNorm 模块的权重使用 AdamW 优化器 (Loshchilov and Hutter, 2017)。对于 AdamW，我们设置其超参数为 β_1 = 0.9, β_2 = 0.95, ε = 10^{-20}, weight_decay = 0.1。对于 Muon，我们设置动量为 0.95，权重衰减为 0.1，并将每个更新矩阵的 RMS 重缩放为 0.18 以复用 AdamW 的学习率。我们在 32T token 上训练 DeepSeek-V4-Flash，与 DeepSeek-V3 一样，我们也采用批量大小调度策略，将批量大小（以 token 计）从小值逐步增加至 75.5M，然后在训练的大部分时间内保持在 75.5M。学习率在前 2000 步线性预热，在训练的大部分时间内维持在 2.7 × 10^{-4}。在训练接近尾声时，我们最终按照余弦调度将学习率衰减至 2.7 × 10^{-5}。训练以 4K 的序列长度开始，我们逐步将训练序列长度扩展至 16K、64K 和 1M。至于稀疏注意力的设置，我们首先在前 1T token 使用稠密注意力进行预热，然后在序列长度为 64K 时引入稀疏注意力，并在剩余训练过程中保持稀疏注意力。在引入注意力稀疏性时，我们首先设置一个短阶段来预热 CSA 中的 lightning indexer，然后在训练的大部分时间内以稀疏注意力训练模型。对于 auxiliary-loss-free 负载均衡，我们设置偏置更新速度为 0.001。对于均衡损失，我们设置其损失权重为 0.0001 以避免单个序列内的极端不平衡。MTP 损失权重在训练的大部分时间内设置为 0.3，在学习率衰减开始时设置为 0.1。

**DeepSeek-V4-Pro。** 除超参数的具体数值外，DeepSeek-V4-Pro 的训练设置与 DeepSeek-V4-Flash 大体一致。我们对大多数参数采用 Muon 优化器，但对嵌入模块、预测头模块以及所有 RMSNorm 模块的权重使用 AdamW 优化器。AdamW 和 Muon 的超参数与 DeepSeek-V4-Flash 相同。我们在 33T token 上训练 DeepSeek-V4-Pro，同样采用批量大小调度策略，最大批量大小为 94.4M token。学习率调度策略与 DeepSeek-V4-Flash 大体相同，但峰值学习率设置为 2.0 × 10^{-4}，终止学习率设置为 2.0 × 10^{-5}。训练同样以 4K 的序列长度开始，长度逐步扩展至 16K、64K 和 1M。与 DeepSeek-V4-Flash 相比，DeepSeek-V4-Pro 以更长的稠密注意力阶段开始，引入稀疏注意力的策略与 DeepSeek-V4-Flash 相同，遵循两阶段训练方法。对于 auxiliary-loss-free 负载均衡，我们设置偏置更新速度为 0.001。对于均衡损失，我们设置其损失权重为 0.0001 以避免单个序列内的极端不平衡。MTP 损失权重在训练的大部分时间内设置为 0.3，在学习率衰减开始时设置为 0.1。

#### 4.2.3 缓解训练不稳定性

训练万亿参数 MoE 模型面临显著的稳定性挑战，DeepSeek-V4 系列也不例外。我们在训练过程中遇到了显著的不稳定性挑战。虽然简单的回滚可以暂时恢复训练状态，但作为长期解决方案并不充分，因为它们无法防止损失尖峰的再次出现。经验上，我们发现尖峰的出现始终与 MoE 层中的异常值相关联，而路由机制本身似乎加剧了这些异常值的出现。因此，我们试图从两个维度解决此问题：打破由路由引起的恶性循环，以及直接抑制异常值。幸运的是，我们发现了两种有效维持训练稳定性的实用技术。虽然对其底层机制的全面理论理解目前仍是一个开放问题，但我们公开分享它们以促进社区的进一步探索。

**Anticipatory Routing。** 我们发现解耦主干网络和路由网络的同步更新显著提升了训练稳定性。因此，在第 t 步，我们使用当前网络参数 θ_t 进行特征计算，但路由索引使用历史网络参数 θ_{t-Δt} 计算并应用。在实践中，为避免两次加载模型参数的开销，我们在第 t − Δt 步提前获取第 t 步的数据。我们"预期地"计算并缓存将在第 t 步使用的路由索引，这也是我们将此方法命名为 Anticipatory Routing 的原因。我们在基础设施层面也进行了大量优化。首先，鉴于预计算路由索引仅需对数据进行一次前向传播，我们精心编排了流水线执行和计算与 Expert Parallelism (EP) 通信的重叠，成功将 Anticipatory Routing 的额外挂钟时间开销限制在约 20%。其次，我们引入了自动检测机制，仅在损失尖峰发生时触发短回滚并激活 Anticipatory Routing；在此模式下运行一段时间后，系统恢复为标准训练。最终，这种动态应用使我们能够以可忽略的总体额外训练开销避免损失尖峰，且不影响模型性能。

**SwiGLU Clamping。** 在先前的文献 (Bello et al., 2017; Riviere et al., 2024) 中，clamping 已被显式用于约束数值范围，从而增强训练稳定性。在我们的实际训练运行中，我们经验性地发现应用 SwiGLU Clamping (OpenAI, 2025) 有效消除了异常值并大幅有助于稳定训练过程，且不影响性能。在 DeepSeek-V4-Flash 和 DeepSeek-V4-Pro 的整个训练过程中，我们将 SwiGLU 的线性分量 clamp 到 [−10, 10] 范围内，同时将门控分量的上限限制为 10。

## 5 后训练

### 5.1 后训练流程

在预训练之后，我们进行了后训练阶段以产出 DeepSeek-V4 系列的最终模型。尽管训练流程在很大程度上沿袭了 DeepSeek-V3.2，但一项关键的方法论替换被实施：混合强化学习（RL）阶段被完全替换为策略蒸馏（On-Policy Distillation, OPD; Gu et al., 2024; Lu and Lab, 2025）。

#### 5.1.1 专家训练

领域专家的开发通过适配 DeepSeek-V3.2 训练流程来进行。具体而言，每个模型依次经过初始微调阶段和后续由领域特定提示与奖励信号引导的强化学习（RL）阶段进行优化。在 RL 阶段，我们实现了组相对策略优化（Group Relative Policy Optimization, GRPO）算法，超参数与我们先前的研究保持密切一致（DeepSeek-AI, 2025; DeepSeek-AI, 2025）。

推理努力。人们普遍认为，模型在推理任务上的表现从根本上受限于所投入的计算量。因此，我们在不同的 RL 配置下训练了不同的专家模型，以促进针对不同推理能力优化的模型的开发。如 Table 2 所示，DeepSeek-V4-Pro 和 DeepSeek-V4-Flash 均支持三种特定的推理努力模式。对于每种模式，我们在 RL 训练期间施加不同的长度惩罚和上下文窗口，从而产生不同长度的推理输出 token。为整合这些不同的推理模式，我们利用由 `<think\>` 和 `</think\>` token 界定的专用响应格式。此外，对于"Think Max"模式，我们在系统提示的开头添加一条特定指令以引导模型的推理过程，如 Table 3 所示。

生成式奖励模型。通常，易于验证的任务可以使用简单的基于规则的验证器或测试用例有效地优化。相比之下，难以验证的任务传统上依赖于基于人类反馈的强化学习（RLHF），这需要大量人工标注来训练标量奖励模型。然而，在 DeepSeek-V4 系列的后训练阶段，我们摒弃了这些传统的基于标量的奖励模型。取而代之的是，为解决难以验证的任务，我们策划了基于评分准则的 RL 数据，并采用生成式奖励模型（Generative Reward Model, GRM）来评估策略轨迹。关键的是，我们直接对 GRM 本身应用 RL 优化。在这一范式下，actor 网络原生地充当 GRM，实现了模型的评估（判断）能力与其标准生成能力的联合优化。通过统一这些角色，模型的内部推理能力固有地融入其评估过程中，从而产生高度鲁棒的评分。此外，这种方法仅用极少量的多样化人工标注即可实现卓越性能，因为模型利用自身的逻辑在复杂任务间进行泛化。

工具调用模式与特殊 Token。与我们之前的版本一致，我们使用专用的 `<think\></think\>` 标签来界定推理路径。在 DeepSeek-V4 系列中，我们引入了新的工具调用模式，采用特殊的"|DSML|" token 并使用基于 XML 的格式进行工具调用。我们的实验表明，XML 格式有效缓解了转义失败问题并减少了工具调用错误，为模型-工具交互提供了更鲁棒的接口。

交错思考。DeepSeek-V3.2 引入了一种上下文管理策略，在工具结果轮次之间保留推理痕迹，但在新用户消息到达时丢弃它们。虽然有效，但这在复杂智能体工作流中仍然造成了不必要的 token 浪费——每个新的用户轮次都会清除所有累积的推理内容，迫使模型从头重建其问题求解状态。利用 DeepSeek-V4 系列扩展的 1M token 上下文窗口，我们进一步优化了这一机制，以最大化交错思考在智能体环境中的有效性：

• 工具调用场景。如 Figure 7(a) 所示，所有推理内容在整个对话过程中被完全保留。与 DeepSeek-V3.2 在每个新用户轮次丢弃思考痕迹不同，DeepSeek-V4 系列在所有轮次中保留完整的推理历史，包括跨用户消息边界。这使得模型能够在长期智能体任务中维持连贯、累积的思维链。

• 通用对话场景。如 Figure 7(b) 所示，原始策略被保留：当新用户消息到达时，先前轮次的推理内容被丢弃，在持久推理痕迹收益有限的场景中保持上下文简洁。

与 DeepSeek-V3.2 一样，通过用户消息模拟工具交互的智能体框架（如 Terminus）可能不会触发工具调用上下文路径，因此可能无法从增强的推理持久性中受益。我们继续推荐此类架构使用非思考模型。

快速指令。在聊天机器人场景中，许多辅助任务（例如判断是否触发网络搜索、意图识别等）必须在生成响应之前执行。传统上，这些任务由一个独立的小模型处理，由于无法复用已有的 KV cache，需要进行冗余的预填充。为克服这一限制，我们引入了快速指令（Quick Instruction）。我们在输入序列中直接附加一组专用特殊 token，每个 token 对应一个特定的辅助任务。通过直接复用已计算的 KV cache，该机制完全避免了冗余预填充，并允许某些任务（如生成搜索查询、判断权限和领域）并行执行。因此，该方法显著降低了用户感知的首 token 时间（TTFT），并消除了维护和迭代额外小模型的工程开销。

#### 5.1.2 策略蒸馏

通过专门的微调和强化学习训练多个领域特定专家后，我们采用多教师策略蒸馏（On-Policy Distillation, OPD; Gu et al. 2024; Lu and Lab 2025）作为将专家能力合并到最终模型的主要技术。OPD 已成为一种有效的后训练范式，能够高效地将领域专家的知识和能力迁移到单一统一模型中。这通过让学生模型从教师模型在其自身生成轨迹上的输出分布中学习来实现。形式化地，给定一组 N 个专家模型 {π_{E1}, π_{E2}, ..., π_{EN}}，OPD 目标函数定义为：

L_OPD(θ) = Σ_{i=1}^{N} w_i · D_KL(π_θ || π_{Ei}).   (29)

在该公式中，w_i 表示分配给每个专家的权重，通常由专家的相对重要性决定。计算逆 KL 损失 D_KL(π_θ || π_{Ei}) 需要从学生模型 π_θ 采样训练轨迹以保持策略学习。其底层逻辑确保统一策略 π_θ 选择性地从与当前任务上下文相关的专业专家学习（例如，在数学推理任务中与数学专家对齐，在编程任务中与编程专家对齐）。通过这一机制，来自物理上不同专家权重的知识通过 logits 级别的对齐被整合到统一的参数空间中，实际上规避了传统权重合并或混合 RL 技术中常遇到的性能退化。在此阶段，采用覆盖各领域的十多个教师模型来蒸馏单一学生模型。

在处理上述 OPD 目标时，先前的工作通常将全词表 KL 损失简化为每个 token 位置处的 token 级 KL 估计，并通过将 sg log π_{Ei}(y_t | x, y_{<t})（sg 表示停止梯度操作）替换为策略损失计算中每个 token 的优势估计来复用 RL 框架。虽然这种方法资源效率高，但会导致梯度估计的高方差，并经常造成训练不稳定。因此，我们在 OPD 中采用全词表 logit 蒸馏。在计算逆 KL 损失时保留完整的 logit 分布可产生更稳定的梯度估计，并确保对教师知识的忠实蒸馏。在以下小节中，我们描述了使全词表 OPD 在大规模上可行的工程努力。

### 5.2 后训练基础设施

我们的后训练基础设施建立在为 DeepSeek-V3.2 开发的可扩展框架之上。具体而言，我们集成了第 3.4 节中描述的相同分布式训练栈和先前引入的用于高效自回归采样的 rollout 引擎。在此基础之上，我们在本工作中引入了以下主要增强。这些设计使得涉及十多个不同教师模型的超长上下文 RL 和 OPD 合并任务的高效执行成为可能，从而大幅加速模型发布的迭代周期。

#### 5.2.1 FP4 量化感知训练

为实现推理加速并减少部署时的内存流量，我们在后训练阶段引入了量化感知训练（Quantization-Aware Training, QAT）（Jacob et al., 2018），使模型（包括教师模型和参考模型）能够适应量化引入的精度损失。我们将 FP4（MXFP4）量化（Rouhani et al., 2023）应用于两个组件：（1）MoE 专家权重，这是 GPU 内存占用的主要来源（OpenAI, 2025）；（2）CSA 索引器中的 Query-Key（QK）路径，其中 QK 激活以 FP4 完全进行缓存、加载和乘法运算，加速了长上下文场景中的注意力分数计算。此外，我们在该 QAT 过程中进一步将索引分数 I_{:,:} 从 FP32 量化为 BF16。这一优化实现了 top-k 选择器 2 倍的加速，同时保持了 KV 条目 99.7% 的召回率。

对于 MoE 专家权重，遵循 QAT 的常见做法，优化器维护的 FP32 主权重首先被量化为 FP4，然后反量化回 FP8 进行计算。值得注意的是，我们的 FP4 到 FP8 反量化是无损的。这是因为 FP8（E4M3）比 FP4（E2M1）多 2 个指数位，提供了更大的动态范围。因此，只要每个 FP8 量化块（128 × 128 分块）内 FP4 子块（1 × 32 分块）的最大和最小缩放因子之比不超过某个阈值，细粒度的缩放信息就可以被 FP8 扩展的动态范围完全吸收。我们通过实验验证了当前权重满足此条件。这使得整个 QAT 流水线可以完全复用现有的 FP8 训练框架而无需任何修改。在反向传播中，梯度是相对于前向传播中相同的 FP8 权重计算的，并直接传播回 FP32 主权重，等价于通过量化操作应用直通估计器（Straight-Through Estimator, STE）。这也避免了对转置权重重新量化的需要。

在 RL 训练的推理和 rollout 阶段（不涉及反向传播），我们直接使用原生 FP4 量化权重而非模拟量化。这确保了采样期间的模型行为与在线部署完全一致，同时减少了内核内存加载以实现实际加速，并显著降低了内存消耗。我们对 CSA 索引器中的 QK 路径进行类似处理。

#### 5.2.2 全词表 OPD 的高效教师调度

我们的框架支持具有有效无限数量教师的全词表策略蒸馏（OPD），每个教师可能包含万亿级参数。为实现这一点，所有教师权重被卸载到集中式分布式存储，并在教师前向传播期间按需加载，采用类 ZeRO 参数分片以缓解 I/O 和 DRAM 压力。此外，对所有教师物化词表大小 |V| > 100k 的 logits 是不可行的，即使转储到磁盘也是如此。我们通过在前向传播期间仅缓存最后一层教师隐藏状态到集中式缓冲区来解决这一问题。在训练时，这些缓存的状态被检索并通过相应的预测头模块以重建完整 logits。这一设计产生的重计算开销可忽略不计，同时完全规避了与显式 logits 物化相关的内存负担。为减轻教师预测头的 GPU 内存占用，我们在数据调度期间按教师索引排序训练样本。这种安排确保每个不同的教师头在每个 mini-batch 中仅加载一次，且任何时刻最多只有一个教师头驻留在设备内存中。所有参数和隐藏状态的加载/卸载操作在后台异步进行，不阻塞关键路径上的计算。最后，教师和学生 logits 之间的精确 KL 散度使用专用 TileLang 内核计算，加速了计算并减少了动态内存分配。

#### 5.2.3 可抢占与容错 Rollout 服务

为最大化 GPU 资源利用率同时实现高优先级任务的快速硬件供给，我们的 GPU 集群采用集群范围的可抢占任务调度器，任何运行中的任务都可能随时被抢占。此外，硬件故障在大规模 GPU 集群中也很常见。为此，我们为 RL/OPD rollout 实现了可抢占和容错的 LLM 生成服务。

具体而言，我们为每个生成请求实现了 token 粒度的预写日志（Write-Ahead Log, WAL）。每当为某个请求生成新 token 时，我们立即将其追加到该请求的 WAL 中。在抢占期间，我们暂停推理引擎并保存未完成请求的 KV cache。在恢复时，我们使用持久化的 WAL 和保存的 KV cache 继续解码。即使发生致命硬件错误，我们也可以使用 WAL 中持久化的 token 重新运行预填充阶段来重建 KV cache。

重要的是，从头重新生成未完成的请求在数学上是不正确的，因为这会引入长度偏差。由于较短的响应更有可能在中断中存活，从头重新生成会使模型在中断发生时更倾向于产生较短的序列。如果推理栈是批量不变且确定性的，这一正确性问题也可以通过使用采样器中伪随机数生成器的一致种子重新生成来解决。然而，这种方法仍然会产生重新运行解码阶段的额外成本，使其效率远低于我们的 token 粒度 WAL 方法。

#### 5.2.4 面向百万 Token 上下文的扩展 RL 框架

我们引入了针对性优化以实现百万 token 序列上的高效 RL 和 OPD。在 rollout 阶段，我们采用可抢占和容错的 rollout 服务，详见第 5.2.3 节。在推理和训练阶段，我们将 rollout 数据格式分解为轻量级元数据和重量级逐 token 字段。在数据调度期间，可以加载整个 rollout 数据的元数据以执行全局洗牌和打包布局计算。重量级逐 token 字段通过共享内存数据加载器加载以消除节点内数据冗余，并在 mini-batch 粒度消费后立即释放，大幅降低 CPU 和 GPU 内存压力。设备上的 mini-batch 数量根据工作负载动态确定，实现计算吞吐量和 I/O 重叠之间的有效权衡。

#### 5.2.5 面向智能体 AI 的沙箱基础设施

为满足后训练和评估期间智能体 AI 的多样化执行需求，我们构建了生产级沙箱平台 DeepSeek Elastic Compute（DSec）。DSec 由三个 Rust 组件组成——API 网关（Apiserver）、每主机代理（Edge）和集群监控器（Watcher）——通过自定义 RPC 协议互连，并在 3FS 分布式文件系统（DeepSeek-AI, 2025）之上水平扩展。在生产中，单个 DSec 集群管理数十万个并发沙箱实例。

DSec 的设计基于四个观察：（1）智能体工作负载高度异构，从轻量级函数调用到具有多样化操作系统和安全要求的完整软件工程流水线；（2）环境镜像数量众多且体积庞大，但必须快速加载并支持迭代定制；（3）高密度部署要求高效的 CPU 和内存利用率；（4）沙箱生命周期必须与 GPU 训练调度协调，包括抢占和基于检查点的恢复。基于这些观察，我们逐一详述 DSec 的四个核心设计。

统一接口背后的四种执行基板。DSec 暴露一个单一的 Python SDK（libdsec），抽象了四种执行基板。函数调用（Function Call）将无状态调用分派到预热容器池，消除冷启动开销。容器（Container）完全兼容 Docker，利用 EROFS（Gao et al., 2019）按需加载实现高效镜像组装。微虚拟机（microVM）基于 Firecracker（Agache et al., 2020）构建，为安全敏感的高密度部署增加 VM 级隔离。全虚拟机（fullVM）基于 QEMU（Bellard, 2005）构建，支持任意客户操作系统。这四种基板共享通用 API 接口——命令执行、文件传输和 TTY 访问——在它们之间切换仅需更改一个参数。

通过分层存储实现快速镜像加载。DSec 通过分层按需加载协调快速启动与大量且不断增长的环境镜像库。对于容器，基础镜像和文件系统提交存储为 3FS 支持的只读 EROFS 层，直接挂载到 overlay lowerdirs 中。我们在挂载时将文件元数据保存在本地磁盘上；同时，数据块按需从 3FS 获取。对于 microVM，DSec 使用 overlaybd（Li et al., 2020）磁盘格式：只读基础层驻留在 3FS 上用于跨实例共享，而写入操作进入本地写时复制层。此类快照可链式连接，支持高效版本控制和毫秒级恢复。

大规模并发下的密度优化。为适应每个集群数十万个沙箱，DSec 解决了两个资源瓶颈。首先，它缓解了虚拟化环境中的重复页缓存占用，并应用内存回收以实现安全的过度分配。其次，它减轻了容器运行时中的自旋锁竞争，从而降低了每个沙箱的 CPU 开销，显著提高了每主机的打包密度。

轨迹日志与抢占安全恢复。DSec 为每个沙箱维护全局有序的轨迹日志，持久记录每次命令调用及其结果。轨迹服务于三个目的：（1）客户端快速推进——当训练任务被抢占时，沙箱资源仍然保留；在恢复时，DSec 为先前完成的命令重放缓存结果，加速任务恢复的同时也防止非幂等操作的重复执行错误；（2）细粒度溯源——每个状态变化的来源和相应结果可追溯；（3）确定性重放——任何历史会话都可以从其轨迹忠实地复现。

### 5.3 标准基准测试评估

#### 5.3.1 评估设置

知识与推理。知识与推理数据集包括 MMLU-Pro（Wang et al., 2024b）、GPQA（Rein et al., 2023）、Human Last Exam（Phan et al., 2025）、Simple-QA Verified（Haas et al., 2025）、Chinese-SimpleQA（He et al., 2024）、LiveCodeBench-v6（Jain et al., 2024）、CodeForces（内部基准）、HMMT 2026 Feb、Apex（Balunović et al., 2025）、Apex Shortlist（Balunović et al., 2025）、IMOAnswerBench（Luong et al., 2025）和 PutnamBench（Tsoukalas et al., 2024）。

在编程方面，我们在 LiveCodeBench-v6 和内部 Codeforces 基准上评估 DeepSeek-V4 系列。对于 Codeforces，我们收集了 14 场 Codeforces Division 1 比赛，包含 114 道题目（2025 年 5 月至 2025 年 11 月）。Elo 评分计算如下。对于每场比赛，我们为每道题目生成 32 个候选解。对于每道题目，我们无放回地采样其中 10 个解，并以随机顺序排列形成提交序列。每个提交由领域专家构建的测试套件评判。已解决题目的得分遵循 OpenAI（2025）的罚分方案：模型获得以相同失败尝试次数解决同一题目的人类参与者的中位分数。这为每个采样的提交序列产生了总比赛分数，然后通过标准 Codeforces 评分系统转换为比赛排名和估计评分。比赛级别的期望评分定义为对每道题目 10 个提交的所有可能随机选择和排列的估计评分的期望。模型的整体评分是所有 14 场比赛的比赛级别期望评分的平均值。

对于推理和知识任务，我们分别将 Non-think、High 和 Max 模式的温度设为 1.0，上下文窗口分别设为 8K、128K 和 384K token。对于数学任务（如 HMMT、IMOAnswerBench、Apex 和 HLE），我们使用以下模板进行评估："{question}\nPlease reason step by step, and put your final answer within \boxed{}." 对于 DeepSeek-V4-Pro-Max 在数学任务上，我们使用以下模板以激发更深层的推理："Solve the following problem. The problem may ask you to prove a statement, or ask for an answer. If finding an answer is required, you should come up with the answer, and your final solution should also be a rigorous proof of that answer being valid.\n\n{question}"。

对于形式化数学任务，我们在 Lean v4.28.0-rc1（Moura and Ullrich, 2021）上以智能体设置进行评估，可访问 Lean 编译器和语义策略搜索引擎，最多运行 500 次工具调用并使用最大推理努力。此外，我们评估了一个计算更密集的流水线，其中首先生成候选自然语言解并通过自验证（Shao et al., 2025）进行过滤，然后将保留的解作为指导提供给形式化智能体以证明相应的 Lean 语句。这一设计利用非形式化推理改善探索，同时通过形式化验证保持严格的正确性。只有当严格验证器 Comparator 在两种设置下都接受提交时，才将其计为正确。

我们留空了 K2.6 和 GLM-5.1 的一些条目，因为它们的 API 过于繁忙，无法返回我们的查询响应。

1M Token 上下文。由于 DeepSeek-V4 系列支持 1M token 上下文，我们通过选择 OpenAI MRCR（OpenAI, 2024b）和 CorpusQA（Lu et al., 2026）作为基准来评估模型在长上下文场景下的性能。我们重新评估了 Claude Opus 4.6 和 Gemini 3.1 Pro 在这些任务上的表现，旨在标准化所有模型的配置。我们未评估 GPT-5.4，因为其 API 未能响应我们的大部分查询。

智能体。智能体数据集包括 Terminal Bench 2.0（Merrill et al., 2026）、SWE-Verified（OpenAI, 2024e）、SWE Multilingual（Yang et al., 2025）、SWE-Pro（Deng et al., 2025）、BrowseComp（Wei et al., 2025）、MCPAtlas 的公开评估集（Bandi et al., 2026）、GDPval-AA（AA, 2025; Patwardhan et al., 2025）和 Tool-Decathlon（Li et al., 2025）。

对于代码智能体任务（SWE-Verified、Terminal-Bench、SWE-Pro、SWE Multilingual），我们使用内部开发的评估框架评估 DeepSeek-V4 系列。该框架提供最小工具集——bash 工具和文件编辑工具。最大交互步数设为 500，最大上下文长度设为 512K token。关于 Terminal-Bench 2.0，我们知悉 GLM-5.1 指出的环境相关问题。尽管如此，为保持一致性，我们报告在原始 Terminal-Bench 2.0 数据集上的性能。在 Terminal-Bench 2.0 Verified 子集上，DeepSeek-V4-Pro 的得分约为 72.0。

对于搜索智能体任务（BrowseComp、HLE w/ tool），我们也使用内部工具链配合 websearch 和 Python 工具，最大交互步数设为 500，最大上下文长度设为 512K token。对于 BrowseComp，我们使用与 DeepSeek-V3.2（DeepSeek-AI, 2025）相同的丢弃所有上下文管理策略。

知识。在通用世界知识评估中，DeepSeek-V4-Pro-Max（DeepSeek-V4-Pro 的最大推理努力模式）在开源大型语言模型中确立了新的最先进水平。如 SimpleQA-Verified 所示，DeepSeek-V4-Pro-Max 以 20 个绝对百分点的优势显著超越所有现有开源基线。尽管取得了这些进步，它目前仍落后于领先的闭源模型 Gemini-3.1-Pro。在教育知识和推理领域，DeepSeek-V4-Pro-Max 在 MMLU-Pro、GPQA 和 HLE 基准上略微优于 Kimi 和 GLM，但仍落后于领先的闭源模型。总体而言，DeepSeek-V4-Pro-Max 标志着增强开源模型世界知识能力的重要里程碑。此外，DeepSeek-V4-Flash 和 DeepSeek-V4-Pro 在知识型任务上存在显著性能差距；这是预料之中的，因为更大的参数量有助于预训练期间的知识保留。值得注意的是，当分配更高的推理努力时，两个模型在知识基准上均表现出提升的结果。

推理。DeepSeek-V4-Pro-Max 在推理基准上超越了所有先前的开源模型，并在多项指标上匹敌最先进的闭源模型，而较小的 DeepSeek-V4-Flash-Max 在代码和数学推理任务上也超越了此前最佳的开源模型 K2.6-Thinking。与此同时，DeepSeek-V4-Pro 和 DeepSeek-V4-Flash 在编程竞赛中表现出色。根据我们的评估，其性能与 GPT-5.4 相当，这是开源模型首次在该任务上匹敌闭源模型。在 Codeforces 排行榜上，DeepSeek-V4-Pro-Max 目前在人类选手中排名第 23 位。DeepSeek-V4 在智能体和计算密集两种设置下的形式化数学任务上也表现出强劲性能。在智能体设置下，它取得了最先进的结果，如 Figure 8 所示，超越了 Seed Prover（Chen et al., 2025）等先前模型。在计算更密集的流水线下，性能进一步提升，超越了包括 Aristotle（Achim et al., 2025）在内的系统，并匹敌该设置下的已知最佳结果。

智能体。DeepSeek-V4 系列在评估中展现出强劲的智能体性能。在代码智能体任务上，DeepSeek-V4-Pro 取得了与 K2.6 和 GLM-5.1 相当的结果，尽管所有这些开源模型仍落后于其闭源对照模型。DeepSeek-V4-Flash 在编程任务上不如 DeepSeek-V4-Pro，特别是在 Terminal Bench 2.0 上。在其他智能体评估中也观察到类似趋势。值得注意的是，DeepSeek-V4-Pro 在 MCPAtlas 和 Toolathlon 上表现良好——这两个评估测试集包含广泛的工具和 MCP 服务——表明我们的模型具有出色的泛化能力，并非仅在内部框架上表现良好。

1M Token 上下文。DeepSeek-V4-Pro 在 MRCR 任务（衡量上下文内检索能力）上优于 Gemini-3.1-Pro，但仍落后于 Claude Opus 4.6。如 Figure 9 所示，检索性能在 128K 上下文窗口内保持高度稳定。虽然超过 128K 标记后性能退化变得可见，但与闭源和开源对照模型相比，模型在 1M token 处的检索能力仍然非常强劲。与 MRCR 不同，CorpusQA 更接近真实场景。评估结果也表明 DeepSeek-V4-Pro 优于 Gemini-3.1-Pro。

推理努力。如 Table 7 所示，在 RL 中采用更长上下文和较低长度惩罚的 Max 模式，在最具挑战性的任务上优于 High 模式。Figure 10 展示了 DeepSeek-V4-Pro、DeepSeek-V4-Flash 和 DeepSeek-V3.2 在代表性推理和智能体任务上的性能与成本比较。通过扩展测试时计算，DeepSeek-V4 系列相比前代模型取得了显著提升。此外，在 HLE 等推理任务上，DeepSeek-V4-Pro 展现出比 DeepSeek-V3.2 更高的 token 效率。

### 5.4 真实世界任务性能

标准化基准测试通常难以捕捉多样化真实世界任务的复杂性，在测试结果与实际用户体验之间造成差距。为弥合这一差距，我们开发了专有的内部指标，优先考虑真实世界使用模式而非传统基准测试。这种方法确保我们的优化转化为切实的收益。我们的评估框架专门针对 DeepSeek API 和聊天机器人的主要用例，使模型性能与实际需求对齐。

#### 5.4.1 中文写作

DeepSeek 的主要用例之一是中文写作。我们对功能写作和创意写作进行了严格评估。Table 12 展示了 DeepSeek-V4-Pro 与 Gemini-3.1-Pro 在功能写作任务上的成对比较。这些任务由常见的日常写作查询组成，提示通常简洁直接。选择 Gemini-3.1-Pro 作为基线，因为它是我们评估中中文写作表现最佳的外部模型。结果表明，DeepSeek-V4-Pro 以 62.7% 对 34.1% 的总体胜率优于基线；这主要是因为 Gemini 偶尔会使其固有风格偏好凌驾于中文写作场景中用户的明确需求之上。

Table 13 展示了创意写作的比较，沿两个轴进行评估：指令遵循和写作质量。与 Gemini-3.1-Pro 相比，DeepSeek-V4-Pro 在指令遵循上取得 60.0% 的胜率，在写作质量上取得 77.5% 的胜率，在指令遵循上表现出边际提升，在写作质量上取得显著增益。尽管 DeepSeek-V4-Pro 在整体用户案例分析中产生了更优结果，但仅限于最具挑战性提示的评估——特别是涉及高复杂度约束或多轮场景的提示——表明 Claude Opus 4.5 仍对 DeepSeek-V4-Pro 保持性能优势。如 Table 14 所示，Claude Opus 4.5 取得 52.0% 的胜率，对比 45.9%。

#### 5.4.2 搜索

搜索增强问答是 DeepSeek 聊天机器人的核心能力。在 DeepSeek 网页端和 App 上，"非思考"模式采用检索增强搜索（Retrieval-Augmented Search, RAG），而"思考"模式利用智能体搜索。

检索增强搜索。我们在客观和主观问答类别中对 DeepSeek-V4-Pro 和 DeepSeek-V3.2 进行了成对评估。如 Table 11 所示，DeepSeek-V4-Pro 以显著优势优于 DeepSeek-V3.2，在两个类别中均展现出一致优势。最显著的增益出现在单值搜索和规划与策略任务上，表明 DeepSeek-V4-Pro 在定位精确事实答案和从检索上下文综合结构化计划方面表现出色。然而，DeepSeek-V3.2 在比较和推荐任务上仍具有相对竞争力，表明 DeepSeek-V4-Pro 在需要对搜索结果进行平衡、多视角推理的场景中仍有改进空间。

智能体搜索。与标准 RAG 不同，智能体搜索使模型能够按查询迭代调用搜索和获取工具，显著提升整体搜索性能。对于 DeepSeek-Chat 中的思考模式，我们优化了智能体搜索功能，以在预定义的"思考预算"内最大化响应准确性。如 Table 9 所示，智能体搜索一致优于 RAG，特别是在复杂任务上。此外，其成本仍然高效，智能体搜索仅比标准 RAG 略贵（见 Table 10）。

#### 5.4.3 白领任务

为严格评估模型在复杂企业生产力场景中的实用性，我们构建了包含 30 项高级中文专业任务的综合套件。这些工作流刻意涵盖高层次的认知需求，包括深度信息分析、综合文档生成和精细文档编辑，横跨 13 个关键行业（如金融、教育、法律和科技）的多样化领域。评估在配备基本工具（包括 Bash 和网络搜索）的内部智能体工具链中进行。

鉴于这些任务的开放性，自动化指标通常无法捕捉高质量响应的细微差别。因此，我们进行了人工评估以比较 DeepSeek-V4-Pro-Max 与 Opus-4.6-Max 的性能。标注者在四个维度上盲评模型输出：

• 任务完成度：核心问题是否被成功解决。
• 指令遵循：对特定约束和指令的遵守程度。
• 内容质量：事实准确性、逻辑连贯性和专业语调。
• 格式美观度：布局可读性和视觉呈现。

如 Figure 11 所示，DeepSeek-V4-Pro-Max 在多样化中文白领任务上优于 Opus-4.6-Max，取得了令人印象深刻的 63% 非损失率，并在分析、生成和编辑任务中展现出一致优势。Figure 12 所示的详细维度分数突出了模型在任务完成度和内容质量方面的主要优势。具体而言，DeepSeek-V4-Pro-Max 通过频繁提供补充见解和自验证步骤，主动预判隐含的用户意图。它在长文本生成方面也表现出色，提供深入、连贯的叙述，而非依赖 Opus-4.6-Max 经常产生的过于简单的要点列表。此外，模型严格遵守正式的专业惯例，如标准化的中文层级编号。然而，在指令遵循方面，它偶尔忽略特定的格式约束，略微落后于 Opus。此外，模型在将大量文本输入浓缩为简洁摘要方面能力不足。最后，在演示幻灯片的整体视觉设计方面，其格式美观度仍有很大改进空间。

#### 5.4.4 代码智能体

为基准测试我们的代码智能体能力，我们从真实内部研发工作负载中策划任务。我们从 50 多名内部工程师处收集了约 200 项具有挑战性的任务，涵盖 PyTorch、CUDA、Rust 和 C++ 等多样化技术栈的功能开发、缺陷修复、重构和诊断。每项任务附带其原始代码仓库、对应的执行环境和人工标注的评分标准；经过严格的质量筛选后，保留了 30 项任务作为评估集。如 Table 8 所示，DeepSeek-V4-Pro 显著优于 Claude Sonnet 4.5，并接近 Claude Opus 4.5 的水平。

在一项调查中，我们询问 DeepSeek 开发者和研究人员（N = 85）——他们均有在日常工作中使用 DeepSeek-V4-Pro 进行智能体编程的经验——DeepSeek-V4-Pro 是否已准备好作为其默认和主要编程模型以替代其他前沿模型，52% 表示肯定，39% 倾向于肯定，不到 9% 表示否定。受访者认为 DeepSeek-V4-Pro 在大多数任务上交付了令人满意的结果，但也指出了细微错误、对模糊提示的误解以及偶尔的过度思考。

## 6 结论、局限性与未来方向

在本工作中，我们发布了 DeepSeek-V4 系列的预览版本，旨在突破超长上下文处理效率壁垒的下一代大型语言模型。通过结合集成 CSA 和 HCA 的混合注意力架构，DeepSeek-V4 系列在长序列效率方面实现了巨大飞跃。这些架构创新，连同广泛的基础设施优化，使得高效原生支持百万 token 上下文成为可能，并为未来的测试时扩展、长期任务以及在线学习等新兴范式奠定了必要基础。评估结果表明，DeepSeek-V4-Pro-Max——DeepSeek-V4-Pro 的最大推理努力模式——重新定义了开源模型的最先进水平。它在知识基准测试上大幅超越此前的开源模型，取得了接近前沿闭源模型的优异推理性能，并展现了具有竞争力的智能体能力。同时，DeepSeek-V4-Flash-Max 在保持高性价比架构的同时，达到了与领先闭源模型相当的推理性能。我们相信，DeepSeek-V4 系列为开源模型开启了百万级上下文的新纪元，并为实现更高效的计算、更大的规模和更强的智能铺平了道路。

在追求极致长上下文效率的过程中，DeepSeek-V4 系列采用了大胆的架构设计。为最小化风险，我们保留了许多经过初步验证的组件和技巧，这些组件和技巧虽然有效，但也使架构相对复杂。在未来的迭代中，我们将开展更全面、更有原则性的研究，将架构精炼至最核心的设计，使其在不牺牲性能的前提下更加优雅。同时，尽管 Anticipatory Routing 和 SwiGLU Clamping 已被证明在缓解训练不稳定性方面有效，但其底层原理仍未被充分理解。我们将积极研究训练稳定性的基础问题，并加强内部指标监控，旨在采用更有原则性和可预测性的方法来实现稳定的大规模训练。

此外，在 MoE 和稀疏注意力架构之外，我们还将积极探索新维度上的模型稀疏性——例如更稀疏的嵌入模块（Cheng et al., 2026）——以在不损害能力的前提下进一步提升计算和内存效率。我们也将持续研究低延迟架构和系统技术，使长上下文部署和交互更加高效响应。此外，我们认识到长期、多轮智能体任务的重要性和实用价值，并将继续在此方向上迭代和探索。我们还在将多模态能力整合到我们的模型中。最后，我们致力于开发更好的数据筛选和合成策略，以持续增强模型在日益广泛的场景和任务中的智能、鲁棒性和实用价值。
